# 08 语义分析：符号表与类型检查

## 本章解决什么问题

语法分析只能判断结构是否像程序。语义分析回答更深的问题：

- 变量使用前是否声明？
- 同名变量指向哪个声明？
- 表达式类型是否正确？
- 函数调用参数数量和类型是否匹配？
- `break` 是否出现在循环里？

## 本章考试能力清单

- 概念题：能解释 binding、environment、symbol table、scope、shadowing、TEnv/VEnv。
- 手算题：能跟踪嵌套作用域环境变化，能按 Tiger 类型规则检查 if/while/for/function/record/array。
- 算法题：能写 `actual_ty`、type compatibility、two-pass 函数/类型声明处理和非法递归类型环判断。
- 英文题：看到 `typing judgment`、`nominal type`、`mutually recursive declarations` 能对应到 Tiger 语义检查。

## 名称解析

名称解析把变量使用连接到对应声明：

```text
let
  var x := 1
in
  x + 2
end
```

使用处的 `x` 应绑定到 `var x := 1`。

如果有嵌套作用域：

```text
var x := 1
let
  var x := 2
in
  x
end
```

内层 `x` 遮蔽外层 `x`。

## Binding、Environment、Symbol Table

| 概念 | 含义 |
|---|---|
| Binding | 名字到含义的映射，如 `x -> int` |
| Environment | 当前可见 binding 的集合 |
| Symbol Table | environment 的实现 |

语义分析通常是一边遍历 AST，一边维护符号表。

## 作用域操作

常见接口：

```text
beginScope()
insert(name, info)
lookup(name)
endScope()
```

`endScope()` 要撤销当前作用域中插入的 binding，恢复外层被遮蔽的名字。

## 符号表实现细节

考试可能不会要求写完整代码，但要理解为什么普通哈希表不够。

一种常见实现是“哈希表 + 栈”：

```text
hash table:
  name -> 当前最内层 binding

stack:
  记录插入顺序，以及每次 beginScope 的 marker
```

操作含义：

| 操作 | 做什么 |
|---|---|
| `beginScope()` | 往栈里压入 scope marker |
| `insert(x, info)` | 把旧的 `x` binding 记到栈上，再让哈希表中 `x` 指向新 binding |
| `lookup(x)` | 直接查哈希表当前 binding |
| `endScope()` | 弹栈直到 marker，逐个恢复旧 binding 或删除当前 binding |

这样 lookup 很快，同时能正确恢复 shadowing。

另一种思路是函数式持久化环境：进入新作用域时创建新 environment，退出时直接丢弃。这更容易解释作用域，但实现和性能取舍不同。

## 类型检查

类型检查为表达式推导类型，并检查操作是否合法。

例：

```text
1 + 2        // int
"a" + "b"    // 如果语言不支持字符串加法，则错误
x < y        // 通常结果为 bool 或 int-like condition
```

## Tiger 类型规则要点

Tiger 的核心类型包括：

- `int`
- `string`
- `unit`，也常在教材里叫 `void`，表示没有有用返回值
- `record`
- `array`
- `nil`
- name type

语义分析常维护两个环境：

| 环境 | 存什么 |
|---|---|
| `TEnv` | 类型名到类型描述，如 `int`、`string`、record、array |
| `VEnv` | 变量名/函数名到类型信息，如变量类型、函数形参和返回类型 |

检查 AST 时常见函数职责：

| 函数 | 职责 |
|---|---|
| `transExp` | 检查表达式并返回其类型 |
| `transVar` | 检查变量访问并返回变量类型 |
| `transDec` | 检查声明并更新环境 |
| `transTy` | 把语法里的类型描述转换为语义类型 |

### Name equivalence

Tiger 中 record/array 通常按名字等价，而不是只看结构。

```tiger
type A = {x:int}
type B = {x:int}
```

`A` 和 `B` 结构相同，但不是同一个类型。

### nil

`nil` 可以和 record 类型兼容，但不能单独确定 record 类型：

```tiger
var r : myRecord := nil   // 可以
var x := nil              // 通常不行，因为无法推断具体 record 类型
```

### 递归类型

递归类型必须通过 record 或 array 间接递归，不能形成纯别名环。

```tiger
type a = b
type b = a        // 非法

type list = {head:int, tail:list}  // 通过 record 合法
```

## Tiger 类型兼容规则总表

| 结构 | 合法条件 | 返回类型 |
|---|---|---|
| 整数/字符串字面量 | 总是合法 | `int` / `string` |
| 变量使用 | 变量在 `VEnv` 中存在 | 变量类型 |
| `a + b`、`a - b`、`a * b`、`a / b` | 两边都是 `int` | `int` |
| `a = b`、`a <> b` | 两边类型兼容；record 可与 `nil` 比较 | `int` 条件值 |
| `a < b` 等比较 | 两边同为 `int` 或同为 `string` | `int` 条件值 |
| `if test then e1 else e2` | `test:int`，`e1/e2` 类型兼容 | 分支公共类型 |
| `if test then e1` | `test:int`，`e1:unit` | `unit` |
| `while test do body` | `test:int`，`body:unit` | `unit` |
| `for i := lo to hi do body` | `lo/hi:int`，`body:unit`，循环变量只读 | `unit` |
| `break` | 必须在循环内部 | `unit` |
| sequence `(e1; e2; ...; en)` | 逐个检查 | `en` 的类型；空序列为 `unit` |
| assignment `var := exp` | 左值和右侧类型兼容 | `unit` |
| function call | 函数存在，参数数量和类型匹配 | 函数返回类型，缺省常为 `unit` |
| record creation | 字段名、数量、顺序/类型符合类型声明 | 对应 record 类型 |
| array creation | size 为 `int`，init 与元素类型兼容 | 对应 array 类型 |
| field access `r.f` | `r` 是 record 且含字段 `f` | 字段类型 |
| subscript `a[i]` | `a` 是 array，`i:int` | 元素类型 |

Tiger 条件通常用 `int` 表示真假：0 为假，非 0 为真。不要把它和 C/Java 的独立 `bool` 类型混为一谈。

## `actual_ty` 与类型兼容

name type 可能只是别名链：

```text
type meters = int
type distance = meters
```

检查兼容前要追到真实类型：

```text
actual_ty(distance) -> actual_ty(meters) -> int
```

伪代码：

```text
actual_ty(t):
  while t is NAME and t.binding is filled:
    t = t.binding
  return t
```

兼容判断：

```text
compatible(a,b):
  a = actual_ty(a)
  b = actual_ty(b)
  if a == b: true
  if a is record and b is nil: true
  if a is nil and b is record: true
  otherwise false
```

record/array 的 `a == b` 在 Tiger 中通常是“同一个类型声明对象”，不是字段结构相同。

## `transExp/transVar/transDec/transTy` 手算流程

语义分析可以理解为四组递归函数：

```text
transExp(venv, tenv, exp, loopDepth) -> type
transVar(venv, tenv, var) -> type
transDec(venv, tenv, decs) -> updated environments
transTy(tenv, tySyntax) -> type
```

关键分情况：

| AST | 检查流程 |
|---|---|
| `VarExp(v)` | 调 `transVar` |
| `OpExp(left, oper, right)` | 分别检查左右，根据运算符要求判断类型 |
| `CallExp(f,args)` | 查函数签名，逐个实参与形参比较 |
| `IfExp(test,then,else)` | test 必须 `int`；有 else 时分支兼容，无 else 时 then 必须 `unit` |
| `WhileExp(test,body)` | test 必须 `int`；body 必须 `unit`；body 内 `loopDepth+1` |
| `BreakExp` | `loopDepth > 0` 才合法 |
| `LetExp(decs,body)` | beginScope，处理 declarations，检查 body，endScope |
| `AssignExp(var,exp)` | var 类型与 exp 兼容；结果 `unit` |

错误恢复策略：发现一个类型错误后，通常返回一个特殊 `error type`，让后续检查继续，避免一个错误引发大量无意义报错。

## Two-pass 处理

互递归函数或类型声明常需要两遍：

1. 第一遍登记名字和类型头部。
2. 第二遍检查定义体。

函数互递归：

```text
function even(n:int):int = if n=0 then 1 else odd(n-1)
function odd(n:int):int = if n=0 then 0 else even(n-1)
```

如果检查 `even` 函数体时还没登记 `odd`，会错误地认为 `odd` 未定义。

类型和函数声明的 two-pass 可按下面模板理解：

```text
类型声明:
  pass 1: 把所有 type name 放进 TEnv，先用占位 name type
  pass 2: 填充每个 name type 的真实定义，并检查非法类型环

函数声明:
  pass 1: 把所有 function header 放进 VEnv，包括参数类型和返回类型
  pass 2: 在扩展后的 VEnv 中检查每个函数体
```

非法类型环的关键判断：如果一串 name type 最终只绕回 name type，而没有经过 record 或 array，就非法。

声明分组也很重要：连续的 type declarations 作为一个互递归组处理，连续的 function declarations 作为一个互递归组处理；变量声明一般按顺序逐条处理。还要检查同一递归组内是否重复定义同名 type/function。

## 例题：类型检查过程

程序：

```tiger
let
  type rec = {x:int}
  var r : rec := nil
  function inc(a:int):int = a + 1
in
  if r = nil then inc(1) else r.x
end
```

检查：

1. 进入 `let` 新作用域。
2. 处理 `type rec = {x:int}`：在 `TEnv` 加入 `rec -> record{x:int}`。
3. 处理 `var r : rec := nil`：`nil` 与 record `rec` 兼容，所以 `r:rec` 加入 `VEnv`。
4. 函数第一遍：加入 `inc : (int) -> int`。
5. 函数第二遍：检查函数体 `a + 1`。参数 `a:int`，`1:int`，加法合法，返回 `int` 与声明一致。
6. 检查 body：`r = nil` 合法，条件类型是 `int`；then 分支 `inc(1):int`；else 分支 `r.x:int`；分支兼容，整个 `if` 为 `int`。
7. 退出作用域，整个 `let` 表达式类型是 `int`。

## 例题：符号表跟踪

程序：

```text
var a:int
let
  var b:string
  var a:string
in
  a
end
```

环境变化：

```text
outer: { a -> int }
enter let: { a -> int, b -> string }
insert inner a: { a -> string, b -> string }  // inner a shadows outer a
lookup a: string
exit let: { a -> int }
```

## 常见误区

- 符号表不是简单 map，因为同名遮蔽需要恢复旧 binding。
- 类型结构一样不一定等价，取决于语言规则。
- `nil` 不是任意类型，只和 record 有特殊兼容关系。
- 递归函数声明不能一边看到一边完整检查，通常要先收集函数头。
- 语义分析不一定生成机器代码，但会为 IR 生成提供类型和访问位置信息。
