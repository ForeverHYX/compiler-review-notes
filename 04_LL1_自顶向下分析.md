# 04 LL(1) 自顶向下分析

## 本章解决什么问题

自顶向下分析从开始符号出发，试图生成输入串。LL(1) 的核心要求是：只看一个 lookahead token，就能唯一决定用哪条产生式。

`LL(1)` 的含义：

- 第一个 `L`：从左到右扫描输入。
- 第二个 `L`：构造最左推导。
- `1`：使用 1 个向前看符号。

## 本章考试能力清单

- 概念题：能解释 LL(1)、lookahead、nullable、FIRST、FOLLOW、predictive parsing table。
- 手算题：能逐轮计算 Nullable/FIRST/FOLLOW，能构造 LL(1) 表，能用 parse stack 模拟输入串。
- 改写题：能消除直接左递归、提左公因子，并判断是否仍有 dangling else。
- 英文题：看到 `conflicting entry`、`epsilon production`、`panic-mode recovery` 能说明含义。

## 本章只学到哪里

这一章只解决“自顶向下怎么不回溯地做语法分析”：

- 从递归下降出发，理解为什么直接试产生式可能需要回溯。
- 用 Nullable/FIRST/FOLLOW 计算每个非终结符在 lookahead 下应选哪条产生式。
- 构造 LL(1) 预测分析表，并用显式栈模拟分析过程。
- 通过提左公因子、消除左递归把部分文法改造成更适合 LL(1) 的形式。
- 在预测分析表空表项处做基本错误恢复。

本章不要求提前掌握 LR 项集、LR 分析表、Yacc 冲突处理；这些是后面自底向上分析的内容。

## 递归下降

每个非终结符写成一个函数。比如：

```text
E -> T E'
E' -> + T E' | epsilon
T -> id
```

可以写成：

```text
parseE():
  parseT()
  parseEPrime()

parseEPrime():
  if lookahead == '+':
    match('+')
    parseT()
    parseEPrime()
  else:
    return
```

如果需要回溯，效率和错误信息都可能变差；LL(1) 通过集合分析避免回溯。

回溯为什么会出现？看这个小文法：

```text
S -> c A d
A -> a b | a
```

输入 `c a d EOF` 时，分析到 `A`，如果先试 `A -> a b`，会成功匹配 `a`，但接下来期待 `b`，实际 lookahead 是 `d`，于是只能退回去改试 `A -> a`。这种“试错”就是回溯。

LL(1) 希望避免这种试错：只看当前 lookahead，就能唯一选择产生式。上面的 `A` 不是 LL(1) 选择点，因为两条候选式的 FIRST 都含有 `a`，看一个 token 分不清。

PPT 里的递归下降代码有两种形态：

- 很简单的文法可以直接用 `switch(tok)` 选择产生式，例如右部第一个符号分别是 `if`、`begin`、`print`。
- 更一般的文法需要先算 FIRST/FOLLOW，再由预测分析表告诉每个函数该走哪个分支。

## Nullable、FIRST、FOLLOW

### Nullable

`Nullable(X)` 表示 `X` 能否推出空串。

### FIRST

`FIRST(alpha)` 是 `alpha` 能推出的串的第一个终结符集合。如果 `alpha` 能推出空串，也包含 `epsilon`。

注意这里的 `alpha` 可以是一串符号，不一定只是一个非终结符。构造 LL(1) 表时真正要用的是每条产生式右部 `alpha` 的 FIRST。

直觉：如果现在要展开 `A -> alpha`，`FIRST(alpha)` 告诉你这条产生式可能以什么 token 开头。

### FOLLOW

`FOLLOW(A)` 是在某个句型中可能紧跟在 `A` 后面的终结符集合。开始符号的 FOLLOW 通常包含 EOF。

FOLLOW 只放终结符或 EOF，不放 `epsilon`。

直觉：如果 `A` 可以消失为 `epsilon`，就要看 `A` 后面可能跟什么。

## FIRST/FOLLOW 手算模板

考试里不要凭直觉一次写完，按固定迭代过程更稳。

### 计算 Nullable

```text
先把所有 Nullable(A) 设为 false
重复直到不变:
  如果有产生式 A -> epsilon，则 Nullable(A)=true
  如果有产生式 A -> X1 X2 ... Xk，且每个 Xi 都 nullable，则 Nullable(A)=true
```

### 计算 FIRST

```text
先把所有 FIRST(A) 设为空集
重复直到不变:
  对每条产生式 A -> X1 X2 ... Xk:
    从 X1 开始向右看
    如果 Xi 是终结符，把 Xi 加入 FIRST(A)，停止
    如果 Xi 是非终结符，把 FIRST(Xi)-{epsilon} 加入 FIRST(A)
    如果 Xi 不 nullable，停止
    如果所有 Xi 都 nullable，把 epsilon 加入 FIRST(A)
```

### 计算 FOLLOW

```text
FOLLOW(S) 加 EOF
重复直到不变:
  对每条产生式 A -> X1 X2 ... Xk:
    对每个非终结符 Xi:
      beta = Xi 后面的串
      把 FIRST(beta)-{epsilon} 加入 FOLLOW(Xi)
      如果 beta nullable 或 beta 为空:
        把 FOLLOW(A) 加入 FOLLOW(Xi)
```

做题时建议画三张表：`Nullable` 表、`FIRST` 表、`FOLLOW` 表。每轮只做集合增加，不要删除元素。

## LL(1) 预测分析表

对于产生式 `A -> alpha`：

1. 对每个 `a in FIRST(alpha)`，把 `A -> alpha` 填入 `M[A,a]`。
2. 如果 `epsilon in FIRST(alpha)`，对每个 `b in FOLLOW(A)`，把 `A -> alpha` 填入 `M[A,b]`。
3. 如果某个格子出现多条产生式，文法不是 LL(1)。

读表时要记住两个考试常见判断：

- 空表项：当前 `A` 在 lookahead `a` 下没有合法展开，语法分析器报告 syntax error。
- 冲突表项：同一个 `M[A,a]` 中能填多条产生式，说明只看一个 lookahead 不够，文法不是 LL(1)。

## LL(1) 判断清单

对同一个非终结符 `A` 的任意两条产生式：

```text
A -> alpha
A -> beta
```

必须满足：

1. `FIRST(alpha)` 和 `FIRST(beta)` 不相交。
2. 如果 `alpha` 可空，则 `FIRST(beta)` 不能和 `FOLLOW(A)` 相交。
3. 如果 `beta` 可空，则 `FIRST(alpha)` 不能和 `FOLLOW(A)` 相交。

这三条的直觉是：看一个 lookahead 时，不能同时有两条产生式可选。

还可以用 PPT 提到的性质快速排除：

- LL(1) 文法一定无二义性。
- LL(1) 文法不能有左递归。
- LL(1) 文法不能有未处理的左公因子。

这些条件是必要条件，不是充分条件。也就是说，有左递归一定不是 LL(1)，但“没有左递归”不保证一定是 LL(1)。

## 表驱动分析过程

维护一个栈和输入：

```text
stack = [EOF, S]
while top != EOF:
  X = top(stack)
  a = lookahead(input)
  if X is terminal:
    match X with a
  else:
    use table M[X,a] to expand X
```

如果表项选择 `A -> Y1 Y2 ... Yn`，栈顶弹出 `A` 后，要按反向把 `Yn ... Y2 Y1` 压栈，这样下一轮栈顶才是最左边的 `Y1`。如果右部是 `epsilon`，只弹出 `A`，什么都不压。

表驱动版本和递归下降版本本质相同：

- 递归下降用函数调用栈保存“后面还要分析什么”。
- 表驱动分析把这件事显式写成 parse stack。

## 消除直接左递归

左递归会让递归下降无限递归。

原文法：

```text
A -> A alpha | beta
```

改写：

```text
A  -> beta A'
A' -> alpha A' | epsilon
```

例：

```text
E -> E + T | T
```

改为：

```text
E  -> T E'
E' -> + T E' | epsilon
```

## 提左公因子

如果多个产生式开头相同，一个 lookahead 无法决定。

原文法：

```text
A -> alpha beta1 | alpha beta2
```

改写：

```text
A  -> alpha A'
A' -> beta1 | beta2
```

提左公因子的目的不是改变语言，而是推迟选择：先匹配共同前缀 `alpha`，等看到后续 token 后，再在 `A'` 里决定走 `beta1` 还是 `beta2`。

## 例题：计算 FIRST/FOLLOW

文法：

```text
E  -> T E'
E' -> + T E' | epsilon
T  -> F T'
T' -> * F T' | epsilon
F  -> ( E ) | id
```

### Nullable

| 非终结符 | 是否 nullable | 原因 |
|---|---|---|
| `E` | 否 | `T` 不可空 |
| `E'` | 是 | 有 `epsilon` 产生式 |
| `T` | 否 | `F` 不可空 |
| `T'` | 是 | 有 `epsilon` 产生式 |
| `F` | 否 | 只能以 `(` 或 `id` 开始 |

### FIRST

```text
FIRST(F)  = { (, id }
FIRST(T') = { *, epsilon }
FIRST(T)  = { (, id }
FIRST(E') = { +, epsilon }
FIRST(E)  = { (, id }
```

上面旧式表达式里如果 `T -> id`，`FIRST(T)` 只有 `{id}`；这里加入括号和乘法，是为了让 FOLLOW 例题完整。

### FOLLOW

```text
FOLLOW(E)  = { EOF, ) }
FOLLOW(E') = { EOF, ) }
FOLLOW(T)  = { +, EOF, ) }
FOLLOW(T') = { +, EOF, ) }
FOLLOW(F)  = { *, +, EOF, ) }
```

关键来源：

- `FOLLOW(E)` 有 `EOF`，因为 `E` 是开始符号。
- `F -> ( E )` 让 `)` 加入 `FOLLOW(E)`。
- `E -> T E'` 中，`FIRST(E')-{epsilon}={+}` 加入 `FOLLOW(T)`；又因为 `E'` 可空，所以 `FOLLOW(E)` 也加入 `FOLLOW(T)`。
- `T -> F T'` 同理让 `*` 加入 `FOLLOW(F)`，并把 `FOLLOW(T)` 传给 `FOLLOW(F)`。

## 例题：LL(1) 表和栈模拟

继续使用上面的表达式文法。预测分析表中非空格子：

| 非终结符 | lookahead | 产生式 |
|---|---|---|
| `E` | `(`, `id` | `E -> T E'` |
| `E'` | `+` | `E' -> + T E'` |
| `E'` | `)`, `EOF` | `E' -> epsilon` |
| `T` | `(`, `id` | `T -> F T'` |
| `T'` | `*` | `T' -> * F T'` |
| `T'` | `+`, `)`, `EOF` | `T' -> epsilon` |
| `F` | `(` | `F -> ( E )` |
| `F` | `id` | `F -> id` |

模拟输入：

```text
id + id * id EOF
```

栈顶写在右边：

| 步骤 | 栈 | 输入 | 动作 |
|---|---|---|---|
| 0 | `EOF E` | `id + id * id EOF` | `E -> T E'` |
| 1 | `EOF E' T` | `id + id * id EOF` | `T -> F T'` |
| 2 | `EOF E' T' F` | `id + id * id EOF` | `F -> id` |
| 3 | `EOF E' T' id` | `id + id * id EOF` | match `id` |
| 4 | `EOF E' T'` | `+ id * id EOF` | `T' -> epsilon` |
| 5 | `EOF E'` | `+ id * id EOF` | `E' -> + T E'` |
| 6 | `EOF E' T +` | `+ id * id EOF` | match `+` |
| 7 | `EOF E' T` | `id * id EOF` | `T -> F T'` |
| 8 | `EOF E' T' F` | `id * id EOF` | `F -> id` |
| 9 | `EOF E' T' id` | `id * id EOF` | match `id` |
| 10 | `EOF E' T'` | `* id EOF` | `T' -> * F T'` |
| 11 | `EOF E' T' F *` | `* id EOF` | match `*` |
| 12 | `EOF E' T' F` | `id EOF` | `F -> id` |
| 13 | `EOF E' T' id` | `id EOF` | match `id` |
| 14 | `EOF E' T'` | `EOF` | `T' -> epsilon` |
| 15 | `EOF E'` | `EOF` | `E' -> epsilon` |
| 16 | `EOF` | `EOF` | accept |

这个表就是考试里“表驱动 LL(1) 分析”的标准写法。

## Panic-Mode Error Recovery

LL parser 查表时如果 `M[A,a]` 为空，就说明当前非终结符 `A` 无法在 lookahead `a` 下展开。简单错误恢复方法是 panic-mode：

```text
把 FOLLOW(A) 作为同步集合 sync(A)
如果 lookahead 不在 sync(A)，丢弃输入 token
如果 lookahead 在 sync(A)，弹出 A，继续分析
```

直觉：既然 `FOLLOW(A)` 是 `A` 后面合法出现的 token，那么遇到这些 token 时可以假装 `A` 这部分出错并结束，尽量恢复到外层结构。

PPT 里对比了三种策略，考试通常只要知道取舍：

| 策略 | 做法 | 优点 | 风险 |
|---|---|---|---|
| raise and quit | 第一个错误就停止 | 最容易实现 | 只能报一个错误，用户体验差 |
| insert token | 假装缺失 token 已出现 | 有时能保持结构 | 不消耗输入，可能无限循环 |
| delete/skip token | 丢弃输入直到 FOLLOW 同步点 | 一定会前进到 EOF，较稳 | 可能跳过一段本来有用的代码 |

所以课程主推的直觉是：空表项处不要盲目重试；用 FOLLOW 集作为同步点，跳到外层还能继续分析的位置。

## 常见误区

- `FIRST(A)` 看的是 A 能生成的串的开头，不是 A 后面能跟什么。
- `FOLLOW(A)` 不包含 epsilon。
- `epsilon in FIRST(alpha)` 时才需要看 FOLLOW。
- 左递归改写会改变树形结构，但不应改变语言。
- LL(1) 不等于所有文法都能改成 LL(1)。
- “LL(1) 但不是 LR(1)”这类判断题为假；LL(1) 文法属于 LR(1) 可分析范围内。
