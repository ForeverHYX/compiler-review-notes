# 18 循环优化

## 本章只学到哪里

这一章回答：

```text
编译器如何先在 CFG 中认出“可优化的循环”，
再把循环里重复做但结果不变、或可用便宜更新维护的计算移出去/改写掉？
```

PPT 主线：

1. loop 的编译器定义，不是“图中有环”就算 loop。
2. dominator、immediate dominator、dominator tree。
3. back edge、natural loop、nested loop、loop-nest tree。
4. loop preheader。
5. loop-invariant computation 与 safe hoisting。

教材补充但考试只需掌握套路：

- induction variable。
- strength reduction。
- array bounds-check elimination。
- loop unrolling。

弱化内容：

- loop fusion、fission、interchange、peeling、tiling、parallelization 只知道名字和大意。
- 不要求写工业级 loop optimizer，也不要求掌握 SSA/Scalar Evolution 的实现。

## 本章考试能力清单

- 概念题：能区分 cycle、loop、natural loop、header、preheader、back edge。
- 判断题：能看出 dominator 必须是 all paths，不是 some path。
- 手算题：给 CFG 能算 Dom 集合、找 back edge、构造 natural loop。
- 优化题：能判断一个赋值是否 loop-invariant，以及是否满足 safe hoisting 三条件。
- 变换题：能做简单 strength reduction、bounds-check elimination 和 unrolling by 2。
- 英文题：看到 `reducible graph`、`loop latch`、`loop preheader`、`loop-invariant hoisting` 能对应到本章定义。

## 为什么循环优化重要

程序的大部分运行时间常常花在循环里。循环体内一条多余指令，如果循环执行一百万次，就可能变成一百万次浪费。

典型循环优化包括：

| 优化 | 目标 |
|---|---|
| loop-invariant hoisting | 把每轮结果相同的计算移到循环外 |
| induction variable reduction | 用更便宜的变量维护循环变化规律 |
| loop unrolling | 减少分支和索引更新开销 |
| loop fusion | 合并相邻循环 |
| loop fission | 拆开一个循环 |
| loop interchange | 交换嵌套循环顺序 |
| loop tiling | 分块提高 cache locality |
| loop parallelization | 把独立迭代并行执行 |

本课重点不是背所有优化名，而是掌握：

```text
先识别循环结构，再证明某个变换保持语义。
```

## 编译器里的 Loop 定义

CFG 中一个 loop 是一个节点集合 `S`，包含一个 header `h`，满足：

1. `S` 中任意节点都能沿控制流回到 `h`。
2. `h` 能到达 `S` 中任意节点。
3. 从 `S` 外进入 `S` 的边只能指向 `h`。

直觉：

```text
header 是循环唯一入口。
循环可以有多个出口，但入口必须唯一。
```

这和日常说的“有环”不同。一个 CFG 里有 cycle，不一定是编译器意义上的 loop。

### Entry、Exit、Header

| 名称 | 含义 |
|---|---|
| loop header | 循环唯一入口节点 |
| loop entry edge | 从循环外进入 header 的边 |
| loop exit edge | 从循环内跳到循环外的边 |
| loop exiting block | 发出 exit edge 的循环内节点 |
| loop exit block | exit edge 指向的循环外节点 |

考试常见判断：

```text
loop 必须有唯一入口，但可以有多个 exits。
```

## Cycle 不一定是 Loop

例子：

```text
a -> b
a -> c
b -> c
c -> b
```

`b,c` 之间有 cycle，但外部节点 `a` 可以直接进入 `b`，也可以直接进入 `c`。

因此没有一个节点能作为唯一 header：

- 若选 `b`，外部边 `a -> c` 直接进入循环内部。
- 若选 `c`，外部边 `a -> b` 直接进入循环内部。

这种图有 cycle，但不是本章可直接优化的自然循环结构。

## Reducible 与 Irreducible Graph

PPT 强调：

- 结构化控制流通常产生 reducible graph。
- `goto` 等非结构化控制流可能产生 irreducible graph。
- 很多循环优化依赖 reducible graph，因为每个 cycle 都有清楚的 header。

考试深度：

```text
reducible graph: 每个循环都有支配循环体的 header。
irreducible graph: 存在没有唯一 header 的 cycle。
```

不要求证明可约图算法，只要会用它解释“为什么有 cycle 不一定能按自然循环优化”。

## Dominator

在 CFG 中，节点 `d` 支配节点 `n`，如果：

```text
从入口节点 s0 到 n 的每一条路径都经过 d
```

记作：

```text
d dominates n
```

关键字是 every path / all paths。

因此判断题：

```text
A node d dominates n if some path from start to n goes through d.
```

是错的，因为 `some path` 太弱。

基本性质：

- 入口节点支配所有可达节点。
- 每个节点支配自己。
- 不可达节点会带来病态情况；做题通常假设 CFG 已删去不可达节点。

## Dominator 迭代算法

小图手算用集合迭代即可。

设所有节点集合为 `N`，入口为 `s0`：

```text
Dom(s0) = {s0}
对 n != s0: Dom(n) = N

重复直到不变:
  Dom(n) = {n} union intersection(Dom(p) for p in pred(n))
```

为什么取交集？

```text
要支配 n，就必须支配 n 的每一个前驱方向来的路径。
只在某一条路径上出现不够。
```

## Dominator 手算例题

CFG：

```text
1 -> 2
2 -> 3, 4
3 -> 5
4 -> 5
5 -> 2, 6
```

初始：

```text
Dom(1) = {1}
Dom(2..6) = {1,2,3,4,5,6}
```

收敛后：

| 节点 | 前驱 | Dom |
|---|---|---|
| 1 | empty | `{1}` |
| 2 | 1,5 | `{1,2}` |
| 3 | 2 | `{1,2,3}` |
| 4 | 2 | `{1,2,4}` |
| 5 | 3,4 | `{1,2,5}` |
| 6 | 5 | `{1,2,5,6}` |

检查：

- `2` 支配 `3/4/5/6`。
- `5` 不支配 `2`，因为入口到 `2` 有路径 `1 -> 2` 不经过 `5`。

## Immediate Dominator 与 Dominator Tree

`idom(n)` 是严格支配 `n` 的节点中离 `n` 最近的那个。

```text
idom(n) dominates n
idom(n) != n
idom(n) 不再支配 n 的其他更近支配者
```

所有 `idom` 边构成 dominator tree：

```text
idom(n) -> n
```

在 dominator tree 中：

- 父节点是 immediate dominator。
- 祖先节点是 dominator。

做题时可以用它检查 Dom 集合有没有算错。

## Back Edge

边：

```text
n -> h
```

如果 `h dominates n`，则这条边是 back edge，`h` 是循环 header。

不要只看箭头“往回画”：

```text
back edge 的定义依赖 dominator，不依赖图画位置。
```

## Natural Loop

对于 back edge：

```text
n -> h
```

它的 natural loop 包含：

- header `h`。
- tail `n`。
- 所有能到达 `n` 且路径中不经过 `h` 的节点。

等价直觉：

```text
从 tail 沿前驱反向收集，直到 header 截止。
```

PPT 还给出性质：

- natural loop 有唯一入口 header。
- header 支配 natural loop 中所有节点。
- 循环中至少有一条路径回到 header。

## Natural Loop 构造算法

已知 back edge `n -> h`：

```text
loop = {h}
worklist = [n]

while worklist not empty:
  p = pop(worklist)
  if p not in loop:
    add p to loop
    add all predecessors of p to worklist
```

手算时通常写成：

```text
loop = {h, n}
从 n 开始反向找前驱，遇到 h 停止扩展
```

例子继续使用前面 CFG。`5 -> 2` 是 back edge，因为 `2` 支配 `5`。

构造：

```text
loop = {2,5}
pred(5) = {3,4} -> 加入 3,4
pred(3) = {2}   -> 2 已在 loop
pred(4) = {2}   -> 2 已在 loop
```

natural loop：

```text
{2,3,4,5}
```

## 多个 Back Edges 与 Nested Loops

一个 header 可以对应多条 back edges。

如果同一个 header 有多个 natural loops，实际优化中常把这些 natural loops 合并成一个 loop。

nested loop：

```text
B 的所有节点都在 A 内，且 A/B header 不同
=> B nested within A
```

定理直觉：

```text
两个 natural loops 如果 header 不同，
要么互不相交，要么一个完全包含另一个。
```

innermost loop 是不再包含其他 loop 的 loop。优化常从 innermost loop 开始，因为它执行最频繁、结构也较小。

## Loop-Nest Tree

PPT 的构造流程：

```text
1. 计算 CFG 的 dominators。
2. 构造 dominator tree。
3. 找所有 back edges 和 natural loops。
4. 对每个 header，把同 header 的 natural loops 合并。
5. 按包含关系构造 loop-nest tree。
```

loop-nest tree 中：

- 每个节点代表一个 loop 或合并后的 loop。
- 父子关系表示嵌套。
- 叶子节点是 innermost loops。
- 整个 procedure body 可看成 pseudo-loop root。

考试一般不会让你完整构造大树，但会问“inner loop / nested loop / same header merge”的概念。

## Loop Preheader

loop preheader 是插在循环外、紧挨 header 之前的基本块。

```text
outside -> preheader -> header -> loop body
```

性质：

- preheader 不属于循环。
- preheader 只有一个 successor，就是 header。
- 所有从循环外到 header 的路径都经过 preheader。

为什么需要 preheader？

```text
loop-invariant hoisting 需要一个安全位置放被外提的计算。
```

如果循环外有多条边直接进 header，先插入 preheader，把外部入口统一到 preheader，再跳到 header。

## Loop-Invariant Computation

loop-invariant computation 是每次执行该计算时结果都一样的计算。

例子：

```text
L0:
  a := x
L1:
  i := i + 1
  b := 7
  t := a + b
  M[i] := t
  if i < N goto L1 else L2
```

在循环 `L1` 中：

| 赋值 | 是否 invariant | 原因 |
|---|---|---|
| `i := i + 1` | 否 | `i` 每轮变化 |
| `b := 7` | 是 | 常量 |
| `t := a + b` | 是 | `a` 在循环外定义，`b` 是 invariant |
| `M[i] := t` | 否 | 地址 `i` 每轮变化，且是内存写 |

## 如何识别 Loop Invariant

对赋值：

```text
x := v1 op v2
```

如果每个操作数 `vi` 满足以下三种情况之一，则该赋值是 loop-invariant：

1. `vi` 是常量。
2. 所有到达此赋值的 `vi` 定义都在循环外。
3. 只有一个循环内定义到达此赋值，且那个定义本身已知是 loop-invariant。

因此可以迭代计算：

```text
先找操作数全是常量/循环外定义的赋值。
再用这些已知 invariant 继续推出新的 invariant。
直到集合不再变化。
```

注意：这里说的是“某个 definition / computation 是否 invariant”，不一定说变量名在整个循环中只有一个值。

## Loop Variant 与回忆卷题型

loop-variant 表示值可能随迭代变化。

回忆卷形式：

```text
i <- 0
j <- i
Loop:
  a <- j + 1
  b <- a * c
  c <- ...
  i <- i + 1
  if ... goto Loop
```

判断：

| 名称 | 分析 |
|---|---|
| `j` | 循环外定义，循环内不改，是 invariant value |
| `a <- j+1` | RHS 不变，所以这个 computation 是 loop-invariant |
| `b <- a*c` | 取决于 `c`，若 `c` 循环内改变则 variant |
| `c <- ...` | 循环内定义，通常 variant，除非题目给出 RHS 不变 |
| `i <- i+1` | basic induction variable，variant |

如果选项只有 `a/b/c/d`，问“不随迭代变化”或 “not all-loop-variant”，通常选 `a`。

## Hoisting

hoisting 是把 loop-invariant computation 移到 preheader。

原来：

```text
L1:
  c := a + b
  d := i * 2
  if i < N goto L1 else L2
```

如果 `a+b` invariant 且外提安全：

```text
preheader:
  c := a + b
L1:
  d := i * 2
  if i < N goto L1 else L2
```

收益：

```text
原来每轮算一次，现在只算一次。
```

## Safe Hoisting 三条件

PPT 给出的核心条件。对 invariant assignment：

```text
d: t <- a op b
```

安全外提需要：

| 条件 | 含义 | 防止什么错误 |
|---|---|---|
| 1. `d` dominates all loop exits where `t` is live-out | 如果循环外要用 `t`，原程序必须保证所有相关出口前都执行过 `d` | 外提让原本可能不赋值的 `t` 被赋新值 |
| 2. there is only one definition of `t` in the loop | 循环内 `t` 只有这一个定义 | 多个定义被外提后改变使用到的版本 |
| 3. `t` is not live-out of the loop preheader | 进入循环前没有还活着的旧 `t` | 外提覆盖第一轮循环中本应使用的旧值 |

再加上通用语义安全：

- 表达式不能有副作用。
- 外提不能提前触发异常，例如除零、空指针、越界。
- 对 load/call 要考虑 alias 和内存副作用；考试一般不深挖。

## 为什么 Invariant 不一定能外提

反例 1：定义不支配相关 exit。

```text
for (...) {
  if (cond) t = a + b
  if (error) break
  use(t)
}
use(t)
```

如果原程序在 `cond` 为假且 `error` 为真时跳出，`t` 没有被本轮赋值。外提前：

```text
t = a + b
```

会改变 loop 后 `use(t)` 看到的值。

反例 2：循环内有多个 `t` 定义。

```text
t = a + b
use(t)
t = c + d
use(t)
```

只外提其中一个定义会改变后续 `use(t)` 的值。

反例 3：`t` 在 preheader live-out。

```text
t := 0
L1:
  M[i] := t
  t := a + b
```

第一轮 `M[i] := t` 应使用旧值 `0`。如果提前执行 `t := a+b`，第一轮就错。

## While 到 Repeat 的直觉

PPT/教材提到一个细节：某些 while loop 中，循环体的语句不一定支配 loop exit，因为可能一开始条件就 false。

为了让外提条件更容易满足，编译器有时会做类似 loop rotation：

```text
if condition:
  repeat:
    body
  until not condition
```

考试不要求画完整变换，但要理解：

```text
外提不仅要看表达式不变，还要看原程序是否保证会执行这个定义。
```

## Induction Variables

归纳变量按固定规律随循环迭代变化。

basic induction variable：

```text
i := i + c
i := i - c
```

其中 `c` 是 loop-invariant。

derived induction variable：

```text
j := i * 4
k := j + base
```

如果 `i` 每轮加 1，`j` 每轮实际加 4，`k` 每轮也加 4。

考试重点：

```text
识别 i 是 basic induction variable，
识别 4*i 或 base+4*i 是 derived induction variable。
```

不要求完整证明 induction-variable family。

## Strength Reduction

strength reduction 把循环里的昂贵运算改成便宜更新。

原循环：

```text
s := 0
i := 0
L1:
  if i >= n goto L2
  j := i * 4
  k := j + a
  x := M[k]
  s := s + x
  i := i + 1
  goto L1
L2:
```

改写：

```text
s := 0
k' := a
limit := a + n * 4
L1:
  if k' >= limit goto L2
  x := M[k']
  s := s + x
  k' := k' + 4
  goto L1
L2:
```

变化：

- 循环内不再每轮算 `i*4`。
- 用 `k' := k' + 4` 维护地址。
- 如果 `i` 除了控制循环和计算地址外没有用途，可以删除 `i`。

## Induction Variable Elimination

strength reduction 后，有些 induction variables 变成 useless。

变量 `i` 可删除的典型条件：

- `i` 在循环出口不活跃。
- `i` 的循环内用途只剩更新自己或参与可由新变量替代的比较。
- 比较条件可改写到同 family 的另一个 induction variable 上。

考试答题模板：

```text
1. 找 basic induction variable i。
2. 找 derived expression base + c*i。
3. 在 preheader 初始化新变量 t = base。
4. 循环内用 t 替代 base+c*i。
5. 每轮 t = t + c。
6. 若 i 不再需要，重写 loop bound 并删除 i。
```

## Array Bounds-Check Elimination

安全语言访问数组时可能生成检查：

```text
0 <= i && i < length(a)
```

循环中每次访问都检查会很贵。如果能证明下标范围安全，就可以删除循环体内检查，或把少量检查移到 preheader。

简单例子：

```text
for i = 0; i < length(a); i++:
  sum := sum + a[i]
```

证明：

```text
i 初始为 0
循环条件保证 i < length(a)
每轮 i := i + 1
length(a) 循环内不变
```

所以每次访问都满足：

```text
0 <= i < length(a)
```

不能删除的情况：

- 循环体可能改变数组对象或长度。
- 下标表达式不是简单归纳变量。
- `i*4` 之类计算可能溢出并影响比较语义。
- 有别名写入会改变 `length(a)` 或相关内存。

考试只要求说明“需要范围证明、归纳变量、循环边界和数组长度不变”。

## Loop Unrolling

loop unrolling 复制循环体，让每轮处理多个原始迭代。

原循环：

```text
for i = 0; i < n; i++:
  sum += a[i]
```

unroll by 2：

```text
for i = 0; i + 1 < n; i += 2:
  sum += a[i]
  sum += a[i+1]

for ; i < n; i++:
  sum += a[i]
```

第二个循环是 epilogue / remainder loop，用来处理余数迭代。

为什么需要它？

```text
如果 n 是奇数，主循环每次处理 2 个元素，最后会剩 1 个。
```

优点：

- 减少分支次数。
- 减少循环变量更新次数。
- 暴露更多局部优化机会。

缺点：

- 增加代码大小。
- 寄存器压力可能变大。
- 不一定总是更快。

## 其他循环优化只记大意

| 优化 | 大意 | 本课深度 |
|---|---|---|
| loop fusion | 合并相邻、迭代范围相同的循环 | 知道名字 |
| loop fission | 把一个大循环拆成多个小循环 | 知道名字 |
| loop interchange | 交换嵌套循环顺序，改善 locality | 知道名字 |
| loop peeling | 剥离前几次迭代，简化主循环 | 知道名字 |
| loop tiling/blocking | 分块处理，提高 cache 命中 | 知道名字 |
| loop parallelization | 证明迭代独立后并行执行 | 知道名字 |

不要把这些当作主线大题准备；真正要会手算的是 dominator/natural loop/LICM。

## 大题模板

给 CFG 做循环优化题时按这个顺序写：

```text
1. 标出入口节点和所有 CFG 边。
2. 用 Dom 迭代算法算每个节点的 dominators。
3. 找所有 back edges: n -> h 且 h dominates n。
4. 对每个 back edge 反向收集 natural loop。
5. 如需外提，先说明 preheader。
6. 找 loop-invariant definitions。
7. 对候选外提语句检查三条件:
   - dominates all relevant exits
   - only one definition in loop
   - target not live-out of preheader
8. 做 strength reduction / unrolling 时写出变换前后代码。
```

写大题时，即使最后代码有小错，前 4 步的 CFG 和 Dom 过程也能拿不少分。

## 判断题高频点

| 说法 | 判断 | 解释 |
|---|---|---|
| 有 cycle 就一定是 loop | 错 | loop 需要唯一 header |
| loop 可以有多个 exit | 对 | 但入口必须唯一 |
| `d` dominates `n` 只要求某条路径经过 `d` | 错 | 必须所有入口到 `n` 的路径都经过 |
| back edge 是 `n -> h` 且 `h` dominates `n` | 对 | 定义依赖 dominator |
| natural loop 的 header 支配 loop 内所有节点 | 对 | 自然循环性质 |
| preheader 属于 loop body | 错 | preheader 在循环外 |
| loop-invariant computation 一定可以 hoist | 错 | 还要满足 safe hoisting 条件 |
| 循环内赋值的变量不可能 invariant | 错 | `b:=7`、`a:=j+1` 可是 invariant computation |
| strength reduction 常把乘法换成加法 | 对 | 典型 `4*i` 改为每轮 `+4` |
| unrolling 一定更快 | 错 | 代码膨胀和寄存器压力可能变差 |

## 常见误区

- 把 dominator 的 all paths 看成 some path。
- 只凭图上“往回画”判断 back edge，不检查支配关系。
- 把 cycle、loop、natural loop 混为一谈。
- 找 natural loop 时从 header 继续反向扩到循环外。
- 只判断 RHS 不变就外提，忘记 safe hoisting 三条件。
- 忘记 preheader 不在循环内。
- strength reduction 后忘记更新 loop bound。
- unrolling by 2 后忘记处理奇数次余数。
