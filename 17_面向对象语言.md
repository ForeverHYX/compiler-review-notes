# 17 面向对象语言

## 本章解决什么问题

面向对象语言给编译器增加了类、继承、字段布局、方法调用和动态派发等问题。核心是：编译器如何把高级的 `obj.method()` 翻译成低层内存访问和函数调用。

PPT 重点：类、单继承字段布局、方法编译、静态/动态派发、vtable。  
弱化：多继承字段图着色、membership test、private、OO 优化等可作为补充。

## 类与对象

类描述对象有哪些字段和方法。对象在内存中通常是一个 record，包含：

- class descriptor 指针。
- 字段值。

典型布局：

```text
object:
| descriptor ptr |
| field 0        |
| field 1        |
| ...            |
```

## 单继承字段布局

如果 `B extends A`，通常把 `A` 的字段放在对象开头，再接 `B` 新增字段。

```text
class A { a }
class B extends A { b }
class C extends B { c }

C object:
| descriptor |
| a          |
| b          |
| c          |
```

好处：如果某个变量静态类型是 `A`，即使运行时对象是 `C`，字段 `a` 的 offset 仍然一样。

### 字段 Offset 例题

假设 word size = 4，descriptor 指针占 1 个 word：

```text
class A { a:int; b:int }
class B extends A { c:int }
```

对象布局：

| offset | A object | B object |
|---:|---|---|
| 0 | descriptor | descriptor |
| 4 | `a` | `a` |
| 8 | `b` | `b` |
| 12 |  | `c` |

如果变量 `x` 的静态类型是 `A`，表达式 `x.b` 总是：

```text
MEM(x + 8)
```

即使运行时 `x` 实际指向 `B` 对象，`b` 的 offset 仍然是 8。

## 方法调用

### Static Dispatch

编译期就知道调用哪个函数：

```text
A_f(obj, args)
```

适合 final method、private method 或已知确切类型。

### Dynamic Dispatch

运行时根据对象实际类决定调用哪个方法。

对象的 descriptor 中保存 method table，也叫 vtable 或 dispatch vector。

```text
obj -> descriptor -> method table -> method slot
```

动态调用：

```text
d = MEM(obj + descriptor_offset)
f = MEM(d + method_slot_offset)
CALL f(obj, args)
```

注意：`self/this` 通常作为隐式第一个参数传入。

## Method Override

子类重写方法时，必须保持同一方法在 vtable 中槽位一致。

```text
A.vtable:
slot 0: A.f
slot 1: A.g

B extends A and overrides f:
slot 0: B.f
slot 1: A.g
slot 2: B.h
```

这样 `x.f()` 只要查固定 slot 0，无需知道 `x` 的具体类。

### 动态派发完整翻译

```text
class A { f(); g(); }
class B extends A { override f(); h(); }
```

vtable：

| slot | A.vtable | B.vtable |
|---:|---|---|
| 0 | `A_f` | `B_f` |
| 1 | `A_g` | `A_g` |
| 2 |  | `B_h` |

源代码：

```text
x.f(3)
```

若 `f` 在 slot 0，底层步骤：

```text
descriptor = MEM(x + 0)
methodPtr  = MEM(descriptor + 0 * wordSize)
CALL methodPtr(x, 3)
```

`x` 作为 `self/this` 隐式第一个参数。若 `x` 静态类型是 `A`、运行时对象是 `B`，descriptor 指向 `B`，所以 slot 0 取到 `B_f`。

## Static Type 与 Runtime Type

```text
A x = new B()
x.f()
```

| 名称 | 含义 |
|---|---|
| static type | 编译期从声明看到的类型，这里是 `A` |
| runtime type / dynamic type | 运行时对象真实类，这里是 `B` |

字段访问通常由 static type 决定 offset；动态方法调用由 runtime type 的 vtable 决定目标函数。

## 类描述符

class descriptor 可能包含：

- 父类 descriptor。
- 方法表。
- 字段布局信息。
- 类名或运行时类型信息。
- GC 类型描述符。

## 多继承补充

多继承会破坏“父类字段 offset 在所有子类中不变”的简单规则。可能需要：

- 多个基类子对象。
- 字段 offset 表。
- 用图着色为不共存字段分配相同 offset。

这部分复杂，若未强调，考前只需掌握问题来源和基本思想。

## Instanceof / Membership Test

判断对象是否属于某类或其子类。方法包括：

- 沿父类链向上查。
- 给类层次编号，利用区间判断。
- 位向量或哈希结构。

核心问题是时间和空间权衡。

例：沿父类链判断 `obj instanceof A`：

```text
d = obj.descriptor
while d != null:
  if d == descriptor(A): return true
  d = d.parent
return false
```

区间编号法适合单继承树：给每个类一个 DFS 区间，`B` 是 `A` 子类当且仅当 `id(B)` 落在 `A` 的子树区间内。优点是查询快；缺点是类层次动态变化时维护困难。

## OO 优化

动态派发有开销。如果编译器能证明运行时类型唯一，就可以 devirtualize，把动态调用变成静态调用，甚至 inline。

## 常见误区

- 静态类型和运行时类型不同：变量声明类型可能是父类，实际对象可能是子类。
- 字段访问通常按静态类型固定 offset；方法调用可能按运行时类型动态查。
- vtable slot 必须稳定，否则父类引用无法正确调用子类重写方法。
- `self/this` 不是魔法变量，底层通常是隐式参数。
- 多继承困难主要来自布局和方法/字段冲突。

## 练习

1. 给类层次，画出每个类的对象字段布局。
2. 给 vtable，写出 `obj.f()` 的底层访问步骤。
3. 判断一个方法调用是 static dispatch 还是 dynamic dispatch。
4. 给子类重写方法，更新 vtable。
5. 解释为什么单继承字段布局容易，多继承字段布局困难。
