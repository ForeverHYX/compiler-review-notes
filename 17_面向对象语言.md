# 17 面向对象语言

## 本章只学到哪里

这一章回答：

```text
class/object/method/inheritance 这些高级语言概念，
如何降到 record layout、descriptor、function call 和 runtime type check？
```

PPT 主线：

1. Object-Tiger 的 class/object/method 语法和 `self`。
2. 单继承：字段 prefix layout、method dispatch/vtable。
3. 多继承：为什么 prefix layout 失效，graph coloring 和 hashing 思路。
4. class membership test：`instanceof`/downcast 如何实现。
5. private fields/methods：主要是 compile-time type-checking。

考试深度：

- 单继承对象布局、vtable、dynamic dispatch 要会手算。
- 多继承、membership test、private/classless/OO optimization 多为判断和概念辨析。
- 不要求实现完整 OO Tiger，也不要求掌握复杂 JIT inline cache。

## 本章考试能力清单

- 概念题：能区分 class/object、field/method、static type/runtime type、static/dynamic dispatch。
- 手算题：能画单继承对象字段 offset，能画 vtable slot，能把 `obj.f(args)` 翻译成 descriptor/vtable/call。
- 辨析题：能说明 override 为什么必须保持 slot 稳定，多继承为什么布局困难。
- 判断题：能判断 graph coloring 直接用于对象布局是否会产生空槽，以及 descriptor indirection 如何避免对象空槽。
- 英文题：看到 `receiver`、`method slot`、`RTTI`、`membership test`、`devirtualization` 能知道它们和动态派发的关系。

## OO 给编译器增加什么问题

面向对象语言常见特性：

| 特性 | 对编译器的问题 |
|---|---|
| class | 如何表示类信息和对象布局 |
| object | 如何在 heap 中布局对象 |
| field | 如何用 offset 访问字段 |
| method | 如何把方法编译成函数 |
| inheritance | 子类如何复用父类字段/方法 |
| overriding | 子类方法如何替换父类方法 |
| polymorphism | 父类引用如何调用子类方法 |
| encapsulation | private/protected/public 如何检查 |

底层核心：

```text
obj.method(args)
```

会变成：

```text
method_function(obj, args)
```

其中 `obj` 是 receiver，也就是 `self/this`。

## Object-Tiger 语法概念

课件中的 Object-Tiger 扩展：

```text
class B extends A {
  var x := 0
  method f(a:int): int = ...
}

new B
obj.f(3)
```

继承规则：

- 子类隐式拥有父类所有 fields/methods。
- methods 可以 override。
- fields 通常不能 override。
- overriding method 必须保持相同参数和返回类型。
- 每个 method 有隐式 `self` 参数，运行时绑定到 receiver object。

`self` 不是魔法：

```text
c.move(60)
```

底层可看成：

```text
move(c, 60)
```

## Class Hierarchy

类层次是继承关系图：

| 情况 | 图结构 |
|---|---|
| single inheritance | tree |
| multiple inheritance | DAG |

单继承容易布局，是因为每个类只有一个 parent；多继承困难，是因为一个类可能同时继承多个 parent，无法把所有 parent 的字段都同时放在对象开头。

## 编译 OO 的三件事

PPT 总结的三个问题：

| 问题 | 含义 |
|---|---|
| field layout | 对象字段在内存中怎么排，字段 offset 是多少 |
| method dispatch | `obj.f()` 到底调用哪个 method instance |
| membership test | `obj instanceof C` 或 downcast 如何测试 |

第 17 章主要围绕这三个问题。

## 对象基本布局

对象通常是 heap record：

```text
object:
offset 0: descriptor pointer
offset 1: field 0
offset 2: field 1
...
```

如果 word size = 4 bytes：

```text
descriptor pointer at byte offset 0
first field         at byte offset 4
second field        at byte offset 8
```

descriptor pointer 指向 class descriptor，里面保存运行时类信息。

## Class Descriptor

class descriptor 可能包含：

- parent class descriptor。
- dispatch vector / vtable。
- field offset 信息。
- runtime type information。
- GC type descriptor。
- membership test 需要的 display/bitmap/hash table。

不要把 descriptor 和 object 混淆：

```text
object 很多，每个实例一个。
descriptor 很少，每个类通常一个。
```

所以某些空槽放在 descriptor 里，比放进每个 object 里更能省空间。

## Single Inheritance：字段 Prefix Layout

单继承字段布局使用 prefixing strategy：

```text
child object layout = parent fields as prefix + child new fields
```

例子：

```text
class A extends Object { var a := 0 }
class B extends A { var b := 0; var c := 0 }
class C extends A { var d := 0 }
class D extends B { var e := 0 }
```

按 word offset，不计 descriptor 或把 descriptor 另算：

| class | fields |
|---|---|
| `A` | `a` |
| `B` | `a,b,c` |
| `C` | `a,d` |
| `D` | `a,b,c,e` |

关键性质：

```text
同一个 inherited field 在所有子类中 offset 相同。
```

比如 `a` 在 `A/B/C/D` 中 offset 都一样。

## 字段 Offset 例题

假设 word size = 4，descriptor pointer 占 1 word：

```text
class A { a:int; b:int }
class B extends A { c:int }
```

对象布局：

| byte offset | A object | B object |
|---:|---|---|
| 0 | descriptor | descriptor |
| 4 | `a` | `a` |
| 8 | `b` | `b` |
| 12 |  | `c` |

如果变量 `x` 静态类型是 `A`，表达式：

```text
x.b
```

可翻译为：

```text
MEM(x + 8)
```

即使运行时 `x` 实际指向 `B` 对象，`b` 的 offset 仍然是 8。

## Method Instance

method instance 像函数一样编译：

```text
ClassName_methodName
```

例如：

```text
Vehicle_move
Truck_move
```

semantic analysis 中：

- variable environment entry 指向 class descriptor。
- class descriptor 指向 parent class 和 method instances 列表。
- method instance 有 machine-code label。

同名 method 可能有多个 method instances：

```text
Vehicle_move
Truck_move   // overrides Vehicle_move
```

## Static Dispatch

static method dispatch 在 compile time 决定目标。

```text
c.f()
```

如果 `c` 的 declared type 是 `C`，编译器：

```text
1. 在 C 中找 f。
2. 找不到就沿 parent 往上找。
3. 找到 A_f，就生成 direct call A_f(c, args)。
```

特点：

- 根据变量静态类型。
- 不根据对象 runtime type。
- 无 runtime lookup 开销。
- 不支持 polymorphic overriding 行为。

例子：

```text
Animal a := new Dog
a.describeSpecies()
```

若 `describeSpecies` 是 static method，则调用 `Animal_describeSpecies`，不是 `Dog_describeSpecies`。

## Dynamic Dispatch 与 VTable

dynamic method dispatch 根据 receiver 的 runtime type 决定调用哪个 method instance。

每个 class descriptor 保存一个 dispatch vector，也叫 vtable。

规则：

- inherited methods 保持相同 slot index。
- overridden methods 在同一 slot 替换函数指针。
- new methods 追加新 slot。

例子：

```text
class A { method f(); method g(); }
class B extends A { override f(); method h(); }
```

vtable：

| slot | A.vtable | B.vtable |
|---:|---|---|
| 0 | `A_f` | `B_f` |
| 1 | `A_g` | `A_g` |
| 2 |  | `B_h` |

为什么 slot 必须稳定？

```text
如果 x 静态类型是 A，编译器只知道 f 在 A 的 slot 0。
当 x 运行时指向 B，也必须从 slot 0 找到 B_f。
```

## Dynamic Dispatch 翻译

源代码：

```text
obj.f(arg1, arg2)
```

如果 `f_index` 是 compile-time constant：

```text
d = MEM(obj + 0)                    // load descriptor pointer
p = MEM(d + f_index * wordSize)     // load method pointer
CALL p(obj, arg1, arg2)             // obj is self/this
```

PPT 成本：

```text
2 memory loads + 1 indirect call
```

动态派发比静态调用慢，但支持 polymorphism。

## Static Type 与 Runtime Type

```text
A x = new B()
x.f()
```

| 名称 | 含义 |
|---|---|
| static type | 编译期从声明看到的类型，这里是 `A` |
| runtime type / dynamic type | 运行时对象真实类，这里是 `B` |

字段访问通常由 static type 决定 offset：

```text
x.a -> fixed offset known from A
```

动态方法调用由 runtime type 的 descriptor/vtable 决定：

```text
x.f() -> B descriptor slot for f may point to B_f
```

## 单继承短总结

| 方面 | 技术 | 性质 |
|---|---|---|
| field layout | prefixing | inherited field offset 稳定 |
| static methods | direct call | 无 runtime overhead |
| dynamic methods | vtable / dispatch vector | 2 loads + indirect call |
| override | same slot replacement | 父类引用能调用子类实现 |

## Multiple Inheritance 为什么难

多继承：

```text
class D extends A, B, C
```

prefixing 失败：

```text
D 不可能同时让 A 的字段在开头，
又让 B 的字段在开头，
又让 C 的字段在开头。
```

因此不能简单沿一条 parent chain 排字段。

## 多继承：Field Layout via Graph Coloring

PPT 给出一种 whole-program graph coloring 思路。

建图：

| 图元素 | OO layout |
|---|---|
| node | distinct field name |
| edge | 两个 fields coexist in same class |
| color | offset / slot |

如果两个字段可能出现在同一个对象里，就连边，不能同 offset。

例子：

```text
class A { a }
class B { b; c }
class C extends A { d }
class D extends A, B, C { e }
```

`D` 中共存：

```text
a,b,c,d,e
```

这些字段之间需要按共存关系加边。着色后，每个 field name 得到一个 color/slot。

## Direct Coloring on Objects 会有空槽

如果直接把 graph-coloring 的 color 当 object field offset：

```text
object slot = field color
```

可能出现：

```text
Class B 只有 b,c，
但 color 1 是别的类字段用的，
于是 B object 的 slot 1 空着。
```

所以判断题：

```text
Global graph coloring directly in multiple inheritance may leave empty slots in objects.
```

应理解为：direct coloring on objects 可能导致对象空槽。
如果题干说“graph coloring 本身直接导致空槽”则表述不严谨；真正关键是“直接把 color 当对象 offset”。

## Coloring on Class Descriptors

改进方案：

```text
objects packed tightly
descriptor stores color -> actual object offset
```

对象：

```text
slot 0: descriptor pointer
slot 1..n: fields compact, no gaps
```

descriptor：

```text
descriptor[color_of_field] = actual object offset
```

访问字段 `f`：

```text
d   = MEM(obj + 0)
off = MEM(d + color_of_f * wordSize)
val = MEM(obj + off)
```

对比：

| 方法 | 对象大小 | 字段访问 |
|---|---|---|
| direct coloring on objects | 可能有空槽 | 1 load，固定 offset |
| coloring on descriptors | 对象紧凑，descriptor 可有空槽 | 3 loads |

descriptor 比 object 少得多，所以把空槽放 descriptor 通常更划算。

## Methods 也可用类似 Coloring

PPT 提到 fields 和 methods 甚至可以共享一个 interference graph：

- node：field names + method names。
- edge：两个 names coexist in same class。
- color：descriptor slot。

这部分标注不考算法细节。考试知道：

```text
多继承下 method lookup 和 field offset 一样会变复杂。
```

## Dynamic Loading 问题

global graph coloring 需要：

```text
编译/链接时知道所有 classes。
```

如果语言支持 dynamic class loading：

```text
运行时还会加载新类。
```

就很难提前做全局一致 coloring。

典型例子：Java/C# 这类支持动态装载或独立编译的系统，不能简单依赖 whole-program layout。

## 多继承：Hashing 方案

PPT 给出 hashing 思路：

每个 class descriptor 存：

| 表 | 内容 |
|---|---|
| `Ktab` | field/method name 或 hashed key |
| `Ftab` | field offset 或 method code address |

访问 field `b`：

```text
d = MEM(obj + 0)
k = MEM(d.Ktab + hash(b))
if k != b: handle collision / fail
off = MEM(d.Ftab + hash(b))
val = MEM(obj + off)
```

考试深度：

- hashing 不依赖全局 graph coloring。
- 适合解决 dynamic linking/loading 布局不固定的问题。
- 访问比单继承固定 offset 慢。

## Membership Test

membership test 判断：

```text
obj belongs to class C or subclass of C?
```

常见语法：

- Java `instanceof`
- C# `is`
- Python `isinstance`
- downcasting 前检查
- exception handler 匹配

## Membership Test 方法 1：Linear Search

沿 parent chain 查：

```text
d = obj.descriptor
while d != nil:
  if d == descriptor(C): return true
  d = d.super
return false
```

复杂度：

```text
O(depth)
```

优点简单；缺点深层继承时慢。

多继承时如果有多个 parents，需要搜索 DAG，不再只是单链。

## Membership Test 方法 2：Class Display

单继承树中可以给每个 class 一个 depth，并在 descriptor 里存 ancestor display：

```text
display[i] = ancestor at depth i
```

判断：

```text
obj instanceof C
<=> obj.descriptor.display[C.depth] == C.descriptor
```

例子：

```text
Object depth 0
Animal depth 1
Mammal depth 2
Dog depth 3
```

`myDog instanceof Animal`：

```text
Dog.display[1] == Animal
```

只需一次比较，`O(1)`。

## Membership Test 方法比较

| 方法 | 时间 | descriptor 空间 | 多继承支持 |
|---|---|---|---|
| linear search | `O(depth)` | parent pointer | 可以扩展到多 parent 搜索 |
| class display | `O(1)` | `O(depth)` array | 多继承较复杂 |
| bit vector | `O(1)` | `O(#classes)` bits | 可支持，但空间大 |
| Cohen encoding | `O(1)` | `O(depth)` array | 主要适合单继承 |

考试只需掌握 linear search 和 class display 的直觉。

## Downcast 和 Runtime Check

父类引用转子类时需要 runtime check：

```text
Animal a = new Dog()
Dog d = (Dog) a
```

如果实际对象不是 `Dog` 或其子类，安全语言应抛异常或报错。

静态向下转换不检查会危险：

```text
把 B 对象当 C 对象访问 C-only field
```

会按错误 offset 取数据。

因此：

- Java/Modula-3 等安全语言会检查。
- C++ 的 `static_cast` 可能不做 runtime check；`dynamic_cast` 会做。

## Private Fields and Methods

privacy 主要是 compile-time concept。

编译器在 type-checking 检查：

```text
当前访问是否允许？
```

如果允许，生成代码与 public access 基本相同：

```text
same field offset
same method call mechanism
```

如果不允许：

```text
compile-time error
```

因此 private 通常不需要 runtime check，也没有额外运行时开销。

## Private 与 Inheritance

PPT 重点：

- private methods 不能被 override；子类同名 method 是新的 unrelated method。
- private fields 在内存布局中仍然被继承；子类对象仍含有父类 private field。
- 子类代码不能直接用名字访问父类 private field。

所以：

```text
layout 机制对 private field 和 public field 基本一样；
access control 是 type checker 的工作。
```

## Classless Languages

虎书提到 classless languages，如一些动态 OO 语言，不一定有固定 class declaration。

编译实现可以用类似 pseudo-class descriptor、hash table、method lookup cache。

考试只需知道：

- class-based language 有 class descriptor。
- classless/dynamic language 也需要某种 runtime descriptor 或 lookup structure。
- 性能优化依然围绕 method lookup。

## OO Optimization

dynamic dispatch 灵活但慢：

```text
descriptor load + method load + indirect call
```

优化目标：

```text
把 dynamic call 变成 static direct call。
```

常见优化：

| 技术 | 思想 |
|---|---|
| devirtualization | 证明 receiver runtime type 唯一，改 direct call |
| inlining | 去虚化后把 method body 内联 |
| final/sealed classes | 类不可继承，方法目标更容易确定 |
| class hierarchy analysis | 根据类层次缩小可能目标 |
| inline cache | JIT 中缓存上次 receiver type 到 method |

考试重点：

```text
能解释 devirtualization 是动态派发转静态调用。
```

## 手算模板：对象布局和调用

给 class hierarchy：

```text
1. 先画继承树。
2. 单继承字段：父类 fields 作为 prefix，子类追加新 fields。
3. descriptor pointer 通常放 offset 0。
4. 计算 byte offset：wordSize * slot。
5. vtable：父类 slots 保持，override 替换，同类新方法追加。
6. obj.f(args)：load descriptor -> load method slot -> indirect call with self。
```

## 例题：B extends A

```text
class A { var a; method f }
class B extends A { var b; override f; method g }
```

假设 word size = 4，descriptor at offset 0。

对象布局：

| byte offset | A object | B object |
|---:|---|---|
| 0 | descriptor | descriptor |
| 4 | `a` | `a` |
| 8 |  | `b` |

vtable：

| slot | A.vtable | B.vtable |
|---:|---|---|
| 0 | `A_f` | `B_f` |
| 1 |  | `B_g` |

动态调用：

```text
x.f()
```

如果 `f` slot 是 0：

```text
d = MEM(x + 0)
p = MEM(d + 0 * wordSize)
CALL p(x)
```

若 `x` 静态类型是 `A`、运行时对象是 `B`，descriptor 指向 `B`，所以调用 `B_f`。

## 判断题高频点

| 说法 | 判断 | 解释 |
|---|---|---|
| 子类对象可以放在父类变量中 | 对 | subtype polymorphism |
| 字段访问通常按 runtime type 查 vtable | 错 | 字段 offset 通常由 static type/layout 决定 |
| dynamic method dispatch 按 runtime type 查 method | 对 | 通过 descriptor/vtable |
| override 可以改变 vtable slot | 错 | 必须保持 slot 稳定 |
| `self/this` 底层通常是隐式参数 | 对 | receiver 传给 method |
| 单继承 prefix layout 保证 inherited field offset 稳定 | 对 | 父类字段在前 |
| 多继承 prefix layout 总能同时满足所有父类 | 错 | 多 parent 冲突 |
| direct graph-coloring object layout may leave empty slots in objects | 对 | color 当 offset 时可能有 gaps |
| descriptor indirection 可让 objects compact | 对 | 空槽转移到 descriptor |
| private access 通常是 runtime check | 错 | 多数 class-based 语言是 compile-time type check |
| private fields 不在对象内存中继承 | 错 | 通常仍在布局中，只是不可命名访问 |

## 常见误区

- 静态类型和运行时类型不同：变量声明类型可能是父类，实际对象可能是子类。
- 字段访问和动态方法调用不是同一机制：字段通常固定 offset，方法可能查 vtable。
- vtable slot 必须稳定，否则父类引用无法正确调用子类重写方法。
- `self/this` 不是特殊硬件机制，底层通常是 method 的隐式第一个参数。
- 多继承困难主要来自布局和 method/field name 共存冲突。
- graph coloring 不是“无代价解决多继承”：需要 whole-program，直接 object coloring 可能有空槽。
- private 是 type-checking 问题，不是字段布局问题。

| method instance | 方法实例 | 某类中声明/重写的具体方法代码 |
| inheritance | 继承 | 子类复用父类 |
| encapsulation | 封装 | 访问控制 |
| polymorphism | 多态 | 父类引用可指向子类对象 |
| superclass | 父类 | 被继承的类 |
| subclass | 子类 | 继承者 |
| subtype | 子类型 | 可作为父类型使用 |
| static type | 静态类型 | 编译期类型 |
| runtime type, dynamic type | 运行时类型 | 对象真实类 |
| single inheritance | 单继承 | 一个父类 |
| multiple inheritance | 多继承 | 多个父类 |
| class descriptor | 类描述符 | 运行时类信息 |
| descriptor pointer | 描述符指针 | object offset 0 常见内容 |
| vtable, virtual table | 虚方法表 | 动态派发表 |
| dispatch vector | 派发向量 | vtable 同义 |
| method slot | 方法槽位 | vtable 中固定 offset |
| receiver | 接收者 | 方法调用的对象 |
| runtime type information, RTTI | 运行时类型信息 | descriptor 中的类信息 |
| static dispatch | 静态派发 | 编译期确定目标 |
| dynamic dispatch | 动态派发 | 运行时查方法 |
| method overriding | 方法重写 | 子类替换父类方法 |
| self, this | 自身对象 | 隐式参数 |
| membership test | 成员关系测试 | `instanceof` / downcast |
| class display | 类 display | O(1) membership test |
| downcast | 向下转型 | 父类到子类，需要检查 |
| access control | 访问控制 | private/protected/public |
| private field | 私有字段 | 布局中存在，名字访问受限 |
| classless language | 无类语言 | 动态 OO，仍需 lookup structure |
| devirtualization | 去虚化 | 动态调用转静态调用 |
