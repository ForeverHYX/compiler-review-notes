# 13 活跃变量分析

## 本章只学到哪里

这一章回答一个后端核心问题：

> 哪些 temporary 的当前值在未来还可能被使用？

这件事主要服务于 register allocation：

- 两个 temporary 如果不会同时 live，可以共用一个寄存器。
- 两个 temporary 如果同时 live，就必须在 interference graph 中连边，后面图着色时不能给同一个颜色。

考试重点不是证明数据流理论，而是能：

1. 从 abstract assembly 或三地址代码写 `use/def/succ`。
2. 用 liveness equations 迭代出 `in/out`。
3. 根据 `live-out` 构造 interference graph。
4. 处理 move 指令的特殊构图规则。
5. 分清 `pred/succ`、`in/out`、forward/backward 的方向。

## 本章解决什么问题

前面 instruction selection 生成了 abstract assembly，它仍然假设有无限多 temporaries。真实机器寄存器有限，所以后面 register allocation 要回答：

```text
哪些 temporary 不能放在同一个寄存器？
```

最常见原因是 live range 重叠：

```text
如果 a 和 b 在某个程序点同时 live，
那么 a 和 b 不能共用一个寄存器。
```

liveness analysis 就是为了找到这些同时活跃关系。

## Dataflow Analysis 总览

PPT 先把 liveness 放进更大的 dataflow analysis 框架。

编译器优化通常分三个粒度：

| 粒度 | 中文 | 范围 | 例子 |
|---|---|---|---|
| local | 局部优化 | 一个 basic block 内 | 局部公共子表达式 |
| intraprocedural / global | 过程内全局优化 | 一个函数/过程 | liveness、available expressions |
| interprocedural / whole-program | 过程间优化 | 多个函数甚至全程序 | call graph、LTO |

dataflow analysis 的通用模式：

1. 在 CFG 上收集 facts。
2. 用这些 facts 支持优化或后端决策。

例子：

| fact | 用途 |
|---|---|
| variable live | register allocation、dead code elimination |
| expression available | common subexpression elimination |
| pointer points-to | alias analysis |
| value range | bounds check、constant folding |

本章只要求 liveness。

## CFG 粒度：Statement vs Basic Block

PPT 本章默认：

```text
one node = one statement
```

但也可以把一个 basic block 作为一个 node：

```text
one node = one basic block
```

两种粒度方程形式一样，只是：

- statement-level 用 `use[n]/def[n]`。
- block-level 用 `useB/defB` 或 `gen/kill`。

考试给每条语句让你算 `in/out` 时，用 statement-level；后续优化实现为了快，常用 basic-block-level。

## Live Variable 的定义

变量 `x` 在程序点或节点 `n` 处 live，意思是：

1. 从 `n` 出发存在一条控制流路径到某个使用 `x` 的节点。
2. 这条路径上在使用 `x` 之前没有重新定义 `x`。

更口语地说：

```text
x 当前这个值，将来可能还会被读。
```

如果当前值将来不会再被读，那么这个值是 dead；保存它没有意义。

### Static vs Dynamic Liveness

真实运行时某条路径可能不会走，但静态编译器通常不知道。

| 概念 | 含义 | 近似方向 |
|---|---|---|
| dynamic liveness | 某次真实执行中会从这里走到使用点 | 真实运行事实 |
| static liveness | CFG 上存在某条可能路径到使用点 | 对 dynamic 的 over-approximation |

如果一个变量 dynamically live，它一定 statically live；反过来不一定。

原因是精确程序行为一般不可判定。PPT 用 `f()` 是否 halt 的例子说明：如果精确判断某值将来是否一定会用，就可能需要解决停机问题。

因此 liveness analysis 是保守近似：

- 可以把实际上 dead 的变量误判为 live，最多浪费寄存器。
- 不能把实际上 live 的变量误判为 dead，否则可能把两个仍需区分的值放进同一寄存器，程序算错。

## Liveness 属于哪类数据流分析

分类：

| 维度 | liveness 的类别 | 原因 |
|---|---|---|
| 方向 | backward analysis | 一个值是否需要，取决于未来后继 |
| may/must | may analysis | 只要存在一条后续路径会用它，就要认为 live |
| 近似 | over-approximation | 宁可多认为 live，不能漏掉 live |

容易错的方向：

```text
out[n] 看 successors 的 in。
in[n] 看本节点 use/def 和 out[n]。
```

不是从前驱往后顺着算。

## Flow Graph 术语

CFG 是 directed graph。

对节点 `n`：

| 术语 | 含义 |
|---|---|
| `succ[n]` | 控制流从 `n` 后可能到达的后继节点 |
| `pred[n]` | 控制流可能流入 `n` 的前驱节点 |
| `in[n]` | 节点入口处 live 的变量集合 |
| `out[n]` | 节点出口处 live 的变量集合 |

如果：

```text
1 -> 2
5 -> 2
5 -> 6
```

则：

```text
succ[5] = {2, 6}
pred[2] = {1, 5}
```

考试易错判断：

> 如果 `a in in[n]`，那么对每个 `m in pred[n]`，都有 `a in out[m]`。

这个方向是对的。不要写成 successor。

## use 与 def

对每个节点 `n`：

- `use[n]`：执行 `n` 之前必须读取的变量。
- `def[n]`：执行 `n` 会定义/写入的变量。

常见规则：

| 语句 | use | def |
|---|---|---|
| `x <- y + z` | `{y,z}` | `{x}` |
| `x <- 1` | `{}` | `{x}` |
| `x <- y` | `{y}` | `{x}` |
| `if x < y goto L` | `{x,y}` | `{}` |
| `goto L` | `{}` | `{}` |
| `return x` | `{x}`，以及调用约定要求 live-out 的寄存器 | `{}` |
| `x <- M[p]` | `{p}` | `{x}` |
| `M[p] <- x` | `{p,x}` | `{}` |
| `x <- call f(a,b)` | 参数相关 temporaries | 返回值和 caller-save clobbers |

注意同名变量：

```text
x <- x + 1
```

这里：

```text
use = {x}
def = {x}
```

右侧读取旧值，左侧定义新值。

## Abstract Assembly 中的 use/def

上一章 `AS_instr` 已经记录了：

| 字段 | liveness 中的含义 |
|---|---|
| `src` | `use` |
| `dst` | `def` |
| `jumps` | 构造 CFG successors |

Tiger 编译器里 flow graph 模块提供：

| 接口 | 作用 |
|---|---|
| `FG_use(n)` | 当前 instruction 使用的 temporaries |
| `FG_def(n)` | 当前 instruction 定义的 temporaries |
| `FG_isMove(n)` | 当前 instruction 是否是纯 move |
| `FG_AssemFlowGraph(il)` | 从 instruction list 构造 flow graph |

构造 CFG 时：

- 普通非跳转指令有顺序边到下一条指令。
- `JUMP` 边指向 jump label。
- `CJUMP` 通常有两个 successors：跳转目标和 fall-through。
- `LABEL` 是可被跳转到的位置。

## 三条 Liveness 规则

PPT 从三条规则推出数据流方程。

### 规则 1：live-in 传播到前驱的 live-out

如果：

```text
a in in[n]
```

那么对每个前驱：

```text
for all m in pred[n], a in out[m]
```

因为如果进入 `n` 前需要 `a`，那从任何前驱走到 `n` 后，也必须在前驱出口保留 `a`。

### 规则 2：use implies live-in

如果：

```text
a in use[n]
```

那么：

```text
a in in[n]
```

因为节点执行前就要读 `a`。

### 规则 3：live-out 且本节点不定义，则 live-in

如果：

```text
a in out[n]
a notin def[n]
```

那么：

```text
a in in[n]
```

如果节点之后还需要 `a`，而节点自己没有产生新的 `a`，那进入节点前就要保留旧的 `a`。

## 数据流方程

把三条规则合起来：

```text
out[n] = union in[s] for all s in succ[n]
in[n]  = use[n] union (out[n] - def[n])
```

逐项解释：

| 方程 | 解释 |
|---|---|
| `out[n] = union in[s]` | 后面任意 successor 需要的变量，在 `n` 出口都要保留 |
| `in[n] = use[n] union ...` | 本节点直接使用的变量一定 live-in |
| `out[n] - def[n]` | 本节点没有重定义的 live-out 变量，要从入口传进来 |

`union` 是因为 liveness 是 may analysis：只要某条后继路径可能使用变量，就要保守认为 live。

## 迭代求 fixed point

标准算法：

```text
for each node n:
  in[n] = {}
  out[n] = {}

repeat until no set changes:
  for each node n:
    old_in  = in[n]
    old_out = out[n]
    out[n] = union in[s] for s in succ[n]
    in[n]  = use[n] union (out[n] - def[n])
```

从空集开始，集合只会单调增加；变量数和节点数有限，所以会收敛。

最终结果叫 least fixed point：

- fixed point：再套方程也不变。
- least：所有满足方程的保守解里最小的那个。

为什么要 least？因为它保守但不过度夸大 live set。更大的解也可能满足方程，但会制造更多不必要干涉边。

## 迭代顺序

方程本身不要求某个固定顺序，但顺序影响收敛速度。

liveness 是 backward analysis，所以更快的手算顺序是：

```text
从后往前扫
先算 out[n]
再算 in[n]
```

PPT 示例中，按正向控制流顺序可能要多轮；按反向顺序通常很快。

考试手算建议：

1. 先写 `succ/use/def` 表。
2. 初始所有 `in/out` 为空。
3. 从最后一行往前迭代。
4. 有回边时继续迭代，直到整张表不再变化。
5. 最后再检查每条边 `n -> s` 是否满足 `out[n]` 包含 `in[s]`。

## Block-Level useB/defB

如果把 basic block 当 node，要先算块级摘要。

设 block：

```text
B = [s1, s2, ..., sk]
```

从后往前扫：

```text
useB = {}
defB = {}

for i = k down to 1:
  useB = use(si) union (useB - def(si))
  defB = def(si) union (defB - use(si))
```

直觉：

- `useB` 是“块入口前必须已经有值”的变量。
- `defB` 是“块内会产生新值”的变量。

算出 `useB/defB` 后，方程和 statement-level 一样：

```text
out[B] = union in[S] for S in succ[B]
in[B]  = useB union (out[B] - defB)
```

本课程考试更常给 statement-level，但 block-level 解释了为什么基本块能加速分析。

## 集合表示和复杂度只需知道什么

PPT 提到两类 set representation：

| 表示 | 适合情况 | 操作 |
|---|---|---|
| bit array | dense sets | union 是 bitwise OR |
| sorted list | sparse sets | union 是 merge sorted lists |

复杂度结论只需定性掌握：

- 最坏情况可能很高。
- 实际中 proper node order、稀疏 live sets、基本块粒度会让它跑得很快。
- 也可对每个 temporary 单独从 use sites 向后追溯到 defs，很多 temporary live range 很短，因此实用。

不要把复杂度作为手算重点。

## 例题 1：顺序代码

代码：

```text
1: a <- 1
2: b <- a + 2
3: c <- b + a
4: return c
```

`use/def/succ`：

| 行 | use | def | succ |
|---|---|---|---|
| 1 | `{}` | `{a}` | `{2}` |
| 2 | `{a}` | `{b}` | `{3}` |
| 3 | `{b,a}` | `{c}` | `{4}` |
| 4 | `{c}` | `{}` | `{}` |

反向算：

| 行 | in | out |
|---|---|---|
| 4 | `{c}` | `{}` |
| 3 | `{a,b}` | `{c}` |
| 2 | `{a}` | `{a,b}` |
| 1 | `{}` | `{a}` |

live ranges：

- `a` 从 1 后到 3 前都要保留。
- `b` 从 2 后到 3 前要保留。
- `c` 从 3 后到 return 前要保留。

因此 `a` 和 `b` 在第 2 行后同时 live，会干涉；`c` 不一定和它们干涉。

## 例题 2：分支 CFG

代码：

```text
1: a <- 1
2: b <- 2
3: if c goto 5
4: a <- b + 1
5: d <- a + b
6: return d
```

`succ/use/def`：

| 行 | use | def | succ |
|---|---|---|---|
| 1 | `{}` | `{a}` | `{2}` |
| 2 | `{}` | `{b}` | `{3}` |
| 3 | `{c}` | `{}` | `{4,5}` |
| 4 | `{b}` | `{a}` | `{5}` |
| 5 | `{a,b}` | `{d}` | `{6}` |
| 6 | `{d}` | `{}` | `{}` |

不动点：

| 行 | in | out |
|---|---|---|
| 1 | `{c}` | `{a,c}` |
| 2 | `{a,c}` | `{a,b,c}` |
| 3 | `{a,b,c}` | `{a,b}` |
| 4 | `{b}` | `{a,b}` |
| 5 | `{a,b}` | `{d}` |
| 6 | `{d}` | `{}` |

关键点：

- 第 3 行有两个 successors，所以 `out[3] = in[4] union in[5]`。
- `c` 在第 3 行被 use，所以 `c in in[3]`，并向前传播到第 1、2 行。

## 例题 3：循环 CFG

代码：

```text
1: i <- 0
2: if i < n goto 3 else 5
3: sum <- sum + i
4: i <- i + 1
   goto 2
5: return sum
```

`succ/use/def`：

| 行 | use | def | succ |
|---|---|---|---|
| 1 | `{}` | `{i}` | `{2}` |
| 2 | `{i,n}` | `{}` | `{3,5}` |
| 3 | `{sum,i}` | `{sum}` | `{4}` |
| 4 | `{i}` | `{i}` | `{2}` |
| 5 | `{sum}` | `{}` | `{}` |

不动点：

| 行 | in | out |
|---|---|---|
| 1 | `{n,sum}` | `{i,n,sum}` |
| 2 | `{i,n,sum}` | `{i,n,sum}` |
| 3 | `{i,n,sum}` | `{i,n,sum}` |
| 4 | `{i,n,sum}` | `{i,n,sum}` |
| 5 | `{sum}` | `{}` |

循环里的回边 `4 -> 2` 会把 `n` 和 `sum` 传播回循环体，所以通常需要多轮迭代。

## 从 Liveness 到 Interference Graph

interference graph：

- 节点：temporary 或变量。
- 无向边：两个值不能分到同一个寄存器。

最常见构造规则：

```text
对每条定义 d 的普通指令 n:
  for each v in liveOut[n]:
    if v != d:
      add edge d-v
```

直觉：指令定义 `d` 后，`d` 的新值要放进某个寄存器；如果出口处 `v` 仍 live，那么 `d` 和 `v` 不能占同一个寄存器。

### Move 指令特殊规则

对 move：

```text
d <- s
```

如果 `d` 和 `s` 后面可以分到同一个寄存器，这条 move 可以删掉。因此不要人为给 `d` 和 `s` 加干涉边。

规则：

```text
for each v in liveOut[n] - {s}:
  if v != d:
    add edge d-v
record move(d,s) as coalescing candidate
```

如果后面某条非 move 指令真的让 `d` 和 `s` 同时 live，仍然会自然产生干涉边。

## Interference Graph 例题

用例题 2 的 `in/out`。

定义点：

| 行 | 定义 | live-out | 新边 |
|---|---|---|---|
| 1 | `a` | `{a,c}` | `a-c` |
| 2 | `b` | `{a,b,c}` | `b-a`, `b-c` |
| 4 | `a` | `{a,b}` | `a-b` |
| 5 | `d` | `{d}` | 无 |

最终边：

```text
a-b
a-c
b-c
```

`d` 只在 return 前后活跃，不和其他变量同时 live。

## zero-length live range

虎书提到一个实现细节：如果某条指令定义了 `d`，但 `d` 不在 live-out 中，看起来 `d` 没有后续用途。

这通常是 dead definition，可以被 dead code elimination 删除。但如果这条指令还有副作用，仍会执行并写某个寄存器。那它写寄存器时不能覆盖其他 live 变量。

因此实现 interference graph 时，定义点本身仍要谨慎处理。考试一般不深挖，只要知道“定义但不用”不等于可以随便覆盖 live register。

## Tiger 编译器中的两阶段

虎书把 Tiger 的 liveness 流程分为两步：

```text
Assem instruction list
  -> Flowgraph
  -> Liveness
  -> Interference graph + move list
```

对应模块：

| 模块 | 输入 | 输出 |
|---|---|---|
| `Flowgraph` | `AS_instrList` | CFG，节点带 `use/def/isMove` |
| `Liveness` | flow graph | `Live_graph {graph, moves}` |

`Live_graph` 里：

- `graph` 是 interference graph。
- `moves` 是 move-related pairs，给后面 coalescing 用。

这一点连接到下一章寄存器分配：liveness 的输出不是最终寄存器，而是寄存器分配要用的约束图。

## 大题手算模板

### 模板 1：算 live-in/live-out

1. 给每条语句编号。
2. 写 `succ`。
3. 写 `use/def`。
4. 初始化 `in/out = {}`。
5. 按反向顺序反复套：

```text
out[n] = union in[s] for s in succ[n]
in[n]  = use[n] union (out[n] - def[n])
```

6. 直到整张表不再变化。
7. 回边/循环一定要多迭代。

### 模板 2：构造 interference graph

1. 为每个 temporary 建一个节点。
2. 找每条有 `def` 的指令。
3. 普通指令：`def d` 与 `liveOut[n] - {d}` 全部连边。
4. Move 指令 `d <- s`：只连 `d` 与 `liveOut[n] - {d,s}`，并记录 move pair。
5. 去掉重复边和自边。
6. 画无向图，或列边集合。

### 模板 3：检查答案

1. 每个 `use[n]` 都必须在 `in[n]`。
2. 每条边 `n -> s` 都应有 `in[s] subset out[n]`。
3. 如果 `x in out[n]` 且 `x notin def[n]`，则 `x in in[n]`。
4. 如果某变量只在某条语句定义后再也不用，通常不会在很多地方 live。
5. 分支节点 `out` 是 union，不是 intersection。

## 常见误区

- liveness 是 backward analysis，不是顺着程序执行方向传播。
- `out[n]` 来自 successors 的 `in`，不是 predecessors。
- 如果 `a in in[n]`，应传播到 `pred[n]` 的 `out`，不是 `succ[n]` 的 `out`。
- 分支处取 union，因为只要有一条路径可能使用就要 live。
- `x <- x + 1` 中 `x` 同时在 `use` 和 `def`。
- `def` 杀死的是旧值；右侧 use 要先算。
- 初始空集不是最终答案，循环回边通常要多轮。
- 干涉图是无向图，不要画成 CFG。
- move 指令不要直接给 source 和 destination 加干涉边，否则会影响后面 coalescing。
- 静态 live 是保守近似，不等于某一次具体运行一定会用到。
- liveness 用于 register allocation；不是 heap allocation。
