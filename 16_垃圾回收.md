# 16 垃圾回收

## 本章只学到哪里

垃圾回收属于 runtime system，但它不是“运行时自己就能解决”的孤立模块。精确 GC 必须依赖编译器提供信息：

```text
哪些寄存器/栈槽是指针？
堆对象内部哪些字段是指针？
在哪些程序点可以安全暂停并扫描 roots？
```

PPT 本章主线：

1. GC 的图模型：roots、reachability、garbage。
2. Mark-and-sweep。
3. Reference counting。
4. Copying collection / Cheney algorithm。
5. Interface to the compiler。

明确弱化：

- generational collection：书上有，PPT 标注不考算法细节。
- incremental collection / Baker algorithm：书上有，PPT 标注不考算法细节。
- 只需要知道 remembered set、write barrier、tricolor abstraction 这些词的作用，不展开证明。

## 本章考试能力清单

- 概念题：能解释 garbage、root set、reachability、live object、pointer map、type descriptor、safe point。
- 手算题：能模拟 mark-and-sweep 堆图、reference counting 计数表、Cheney copying 的 `scan/next`。
- 判断题：能判断 reference counting 不能回收循环垃圾；copying collection 能消除外部碎片但不省内存。
- 选择题：能区分 mark-sweep、reference counting、copying collection 的优缺点。
- 英文题：看到 `from-space/to-space`、`forwarding pointer`、`remembered set`、`tricolor abstraction` 能说出用途。

## 为什么需要 GC

堆上对象不像栈帧一样自动按 LIFO 消失：

```text
record / array / object / closure
```

可能在创建函数返回后仍然被其他指针引用，也可能早就不可达但还占着内存。

垃圾回收做的事：

```text
自动找出从程序当前 roots 不可达的堆对象，并回收它们占用的空间。
```

注意：GC 回收的是 heap objects，不是普通栈帧局部变量。栈帧随着函数返回自然释放。

## 什么是 Garbage

理想定义：

```text
以后永远不会再被程序使用的对象。
```

这个定义太强，静态/运行时一般无法精确判断，因为它可能涉及程序未来路径、停机等不可判定问题。

实际 GC 使用可达性近似：

```text
roots 出发沿指针边可达的对象 = 保留
roots 出发不可达的对象 = 可安全回收
```

图模型：

```text
roots: registers, stack slots, globals
          |
          v
reachable heap records = live for GC
unreachable records    = garbage
```

这里的 `live` 是 GC 意义上的可达，不要和第 13 章 variable liveness 混淆。

## Root Set

root set 是 GC 遍历对象图的起点。

常见 roots：

| root 来源 | 例子 |
|---|---|
| registers | 当前寄存器里保存的对象指针 |
| stack slots | 当前函数和调用链上各栈帧里的指针局部变量/参数 |
| globals/static data | 全局指针变量、静态对象引用 |
| runtime internal roots | 运行库维护的全局结构 |

关键问题：

```text
一个机器字到底是 int 还是 pointer？
```

精确 GC 不能乱猜：

- 把整数当指针：可能错误保留垃圾。
- 把指针当整数：可能回收活对象，程序崩溃。

所以后面需要 pointer map 和 type descriptor。

## Freelist

mark-and-sweep 和 reference counting 常需要 freelist。

freelist 是：

```text
空闲 heap blocks 组成的链表。
```

分配时从 freelist 取块；回收时把垃圾块放回 freelist。

如果对象大小不同，可以有多个 freelist：

```text
freelist[1], freelist[2], ..., freelist[k]
```

这样分配大小为 `i` 的对象时，不必在一个长链表里搜索很久。

## GC 技术总览

| 技术 | 基本思想 | 什么时候回收 | 关键弱点 |
|---|---|---|---|
| reference counting | 每个对象记录引用数 | 引用数变 0 时立即回收 | 循环垃圾、更新开销 |
| mark-and-sweep | 从 roots 标记可达对象，再扫描堆 | 内存不足时 stop-the-world | 碎片、扫描整个堆 |
| copying collection | 复制可达对象到新空间，丢弃旧空间 | from-space 满时 stop-the-world | 需要额外空间、移动对象 |

PPT 把 reference counting 也放在 GC 技术里，但它不是 tracing collector。mark-sweep/copying 都属于 tracing：先从 roots 找 reachable objects。

## Mark-and-Sweep

两阶段：

```text
Mark:
  从 roots 出发 DFS/BFS 遍历对象图，标记所有可达 records。

Sweep:
  线性扫描整个 heap。
  marked objects: 清除 mark bit，保留。
  unmarked objects: 放入 freelist。
```

伪代码：

```text
foreach root v:
  DFS(v)

p = heap_start
while p < heap_end:
  if record p is marked:
    unmark p
  else:
    add p to freelist
  p = p + size(p)
```

stop-the-world：

```text
GC 期间程序暂停；回收结束后程序继续。
```

## Mark 阶段细节

DFS 伪代码：

```text
DFS(x):
  if x is a pointer into the heap:
    if record x is not marked:
      mark x
      for each field fi of record x:
        DFS(x.fi)
```

注意 `x` 的说法：

- `x` 的值是某个 record 的地址。
- “record x”是指 `x` 指向的那个 heap record。
- 只有字段是 pointer 时才继续 DFS；整数常量不能当地址跟踪。

## Mark-and-Sweep 手算模板

题目给 heap graph：

```text
1. 画出 roots。
2. 从每个 root 沿指针边遍历。
3. 写 marked set。
4. heap 中不在 marked set 的对象就是 sweep 回收对象。
5. 若存在不可达环，mark-sweep 仍能回收。
```

例子：

```text
roots: r1 -> A, r2 -> D
A -> B
B -> C
D -> B
E -> F
F -> E
G -> none
```

Mark 阶段：

| 起点 | 可达对象 |
|---|---|
| `r1 -> A` | `A,B,C` |
| `r2 -> D` | `D,B,C` |

最终 marked：

```text
{A,B,C,D}
```

Sweep 阶段：

| 对象 | marked? | 处理 |
|---|---|---|
| A | 是 | 清 mark bit，保留 |
| B | 是 | 清 mark bit，保留 |
| C | 是 | 清 mark bit，保留 |
| D | 是 | 清 mark bit，保留 |
| E | 否 | 回收到 freelist |
| F | 否 | 回收到 freelist |
| G | 否 | 回收到 freelist |

`E/F` 虽然互相引用形成环，但从 roots 不可达，所以 mark-and-sweep 能回收它们。

## Mark-and-Sweep 成本和问题

PPT 成本公式：

```text
cost = c1 * R + c2 * H
```

其中：

| 符号 | 含义 |
|---|---|
| `R` | reachable data 的大小 |
| `H` | heap 总大小 |
| `c1` | mark 每个 reachable word 的成本 |
| `c2` | sweep 每个 heap word 的成本 |

每回收 `H-R` 个 word，摊还成本大致：

```text
(c1 * R + c2 * H) / (H - R)
```

直觉：

- 如果 `R` 接近 `H`，说明垃圾很少，每次 GC 回收空间少，成本很高。
- 如果 `H` 比 `R` 大很多，回收成本摊薄。

优点：

- 能回收循环垃圾。
- 不移动对象，对外部/native 指针更友好。
- 实现概念直观。

缺点：

- stop-the-world。
- sweep 要扫描整个堆。
- 会产生外部碎片。
- mark DFS 需要栈或特殊技巧。

## Explicit Stack 与 Pointer Reversal

递归 DFS 可能占用很深调用栈。最坏情况下，指针链长度接近 heap 大小，GC 自己的栈可能爆。

改进：

| 方法 | 思想 | 考试深度 |
|---|---|---|
| explicit stack | 自己维护一个栈保存待访问对象 | 了解 |
| pointer reversal | 临时反转对象字段，用对象自身记录回溯路径 | 了解，不要求手写算法 |

Pointer reversal 的关键直觉：

```text
DFS 回溯路径本来存在栈里；
现在临时写进对象字段里；
回溯时再恢复字段。
```

它省额外空间，但实现复杂。

## Memory Fragmentation

mark-and-sweep 不移动 live objects，所以回收后可能出现：

```text
live | free | live | free | live | free
```

即使总空闲空间足够，也可能找不到一个足够大的连续块，这叫 external fragmentation。

这正是 copying collection 的一个优势：把 live objects 复制到连续空间，天然 compaction。

## Reference Counting

每个对象维护一个 reference count：

```text
rc(x) = 当前有多少指针指向 x
```

规则：

- 新引用指向对象：计数加一。
- 引用不再指向对象：计数减一。
- 计数变零：对象不可达，可立即回收到 freelist。

赋值 `x.fi <- p` 需要做：

```text
old = x.fi
dec rc(old)
if rc(old) == 0: free old
x.fi = p
inc rc(p)
```

实际顺序要避免异常和自赋值问题，考试只需理解“旧目标减、新目标加”。

## Reference Counting 手算模板

```text
1. 列每个 heap object 的 rc。
2. root 指向对象也算引用。
3. object field 指向对象也算引用。
4. 每条赋值：旧边删除，目标 rc--；新边加入，目标 rc++。
5. rc==0 的对象回收，并递归减少它指向对象的 rc。
```

例子：

```text
p = new A
q = new B
p.f = q
q.g = p
p = nil
q = nil
```

计数表：

| 步骤 | rc(A) | rc(B) | 原因 |
|---|---:|---:|---|
| `p=new A` | 1 | 0 | root `p` 指向 A |
| `q=new B` | 1 | 1 | root `q` 指向 B |
| `p.f=q` | 1 | 2 | A 的字段指向 B |
| `q.g=p` | 2 | 2 | B 的字段指向 A |
| `p=nil` | 1 | 2 | root `p` 移除 |
| `q=nil` | 1 | 1 | root `q` 移除 |

最后 A/B 从 roots 不可达，但计数都为 1。

## Reference Counting 优缺点

优点：

- 回收及时，计数为 0 可立刻释放。
- GC 工作与程序执行交错，暂停时间小。
- 可与手动内存管理共存。
- freed cells 可立即复用。

缺点：

- 不能回收循环垃圾。
- 每次指针赋值都要维护计数，开销大。
- 多线程下计数更新还可能需要同步。

判断题：

```text
Reference counting can reclaim all unreachable objects.
```

False。循环垃圾不可达但计数非零。

常见补救：

- 大部分时候用 reference counting。
- 偶尔跑 tracing GC，例如 mark-sweep，清理循环。

这只需了解。

## Copying Collection

把 heap 分成两个 semi-spaces：

| 空间 | 含义 |
|---|---|
| from-space | 程序当前分配对象的空间 |
| to-space | GC 时复制 live objects 的空空间 |

当 from-space 满：

```text
1. 从 roots 出发复制所有 reachable objects 到 to-space。
2. 更新 roots 和对象字段，让它们指向 to-space 中的新地址。
3. 丢弃整个 from-space。
4. from-space / to-space 角色互换。
```

复制不是简单 `memcpy`。必须：

- 保持对象图结构。
- 更新所有指针。
- 同一个对象只复制一次。

## Copying Collection 优缺点

优点：

| 优点 | 解释 |
|---|---|
| fast allocation | 分配通常只是 `p=next; next=next+n` |
| no external fragmentation | live objects 被复制到连续 to-space |
| cost depends on live data | 不扫描死对象，死对象留在 from-space 被整块丢弃 |
| can collect cycles | 基于 reachability，不看 reference count |

缺点：

| 缺点 | 解释 |
|---|---|
| memory overhead | 需要 from-space/to-space，通常一半空间闲置 |
| moves objects | 必须更新所有指针 |
| needs precise type info | 必须知道哪些字段是 pointer |
| stop-the-world | 基础 copying GC 也会暂停程序 |
| locality may be poor | Cheney BFS 顺序不一定符合程序访问顺序 |

选择题：

```text
不属于 copying collection 优点的是 save memory。
```

因为 copying collection 通常更耗内存，不是省内存。

## Forwarding Pointer

copying GC 需要解决：

```text
同一个旧对象被多个指针指到，不能复制多份。
```

方法：forwarding pointer。

当对象 `p` 从 from-space 复制到 to-space 后：

```text
在旧对象 p 的第一个字段 p.f1 写入新地址。
```

后续如果又遇到旧对象 `p`：

```text
检查 p.f1 是否指向 to-space。
如果是，说明已经复制过，直接返回 p.f1。
```

作用：

1. 找到对象的新位置。
2. 避免重复复制。
3. 让所有 roots/fields 都更新到同一个新对象。

## Cheney Algorithm

Cheney 算法用 to-space 本身作为 BFS 队列，不需要额外递归栈。

to-space 被分成三段：

```text
| copied and scanned | copied only | empty |
^start               ^scan         ^next
```

含义：

| 区域 | 含义 |
|---|---|
| start 到 scan | 已复制，且内部指针字段已处理 |
| scan 到 next | 已复制，但内部指针字段还没处理，作为队列 |
| next 之后 | 空闲空间 |

伪代码：

```text
scan = next = beginning of to-space

for each root r:
  r = Forward(r)

while scan < next:
  for each pointer field fi of record at scan:
    scan.fi = Forward(scan.fi)
  scan = scan + size(record at scan)
```

终止条件：

```text
scan == next
```

表示队列为空，所有 copied objects 都已经 scanned。

## Cheney 手算例题

from-space：

```text
roots: r -> A
A fields: [B, C]
B fields: [C]
C fields: []
D fields: []    // 不可达
```

to-space 初始空。

| 步骤 | scan | next | 动作 | to-space |
|---|---|---|---|---|
| 1 | start | start | Forward root `A`，复制 `A'`，root 改指 `A'` | `A'` |
| 2 | `A'` | after `A'` | 扫描 `A'`，复制 `B'`、`C'`，更新字段 | `A',B',C'` |
| 3 | `B'` | after `C'` | 扫描 `B'`，发现 `C` 已有 forwarding pointer，字段改指 `C'` | `A',B',C'` |
| 4 | `C'` | after `C'` | `C'` 无指针字段，推进 scan | `A',B',C'` |
| 5 | scan==next | after `C'` | 结束，丢弃 from-space | `A',B',C'` |

`D` 没被复制，因此自然回收。

考试画图时要写清：

- roots 更新到新对象。
- 对象字段也更新到新对象。
- forwarding pointer 只存在于旧对象里，是 GC 过程中的临时信息。

## Locality：BFS vs DFS

Cheney 是 BFS copying：

- 同一层对象可能相邻。
- parent 和 child 未必相邻。

DFS copying 往往让 parent 和 child 更接近，局部性更好，但需要栈或更复杂算法。

PPT 中 hybrid algorithm 了解即可：

```text
部分 depth-first，部分 breadth-first，用来改善 locality。
```

考试只需会说：

- copying GC 可改善 fragmentation。
- Cheney BFS 不一定改善程序访问 locality。
- locality 优化不是本章主手算。

## Interface to the Compiler

PPT 最后强调：GC 虽然在 runtime，但编译器必须合作。

编译器需要支持：

| 支持 | 用途 |
|---|---|
| allocation code | 生成 record/array/object 分配代码 |
| data layout descriptors | 描述对象内部哪些字段是指针 |
| pointer maps / stack maps | 描述 safe point 处 roots 在哪些栈槽/寄存器 |
| pointer tags on temps | 临时变量一路携带 pointer/non-pointer 信息 |
| safe points | 只在这些点暂停并扫描 roots |
| barriers | 分代/增量 GC 需要 read/write barrier |

## Fast Allocation

copying collection 的 fast allocation：

```text
if next + N >= limit:
  call GC
result = next
next = next + N
```

优化点：

- inline expand allocation fast path。
- 让 `next` 直接进入最终保存新对象指针的寄存器，减少 copy。
- 多个连续 allocation 可以合并一次 bounds check。

考试深度：

```text
知道 copying GC 支持 very fast allocation；
不用手写复杂优化代码。
```

## Pointer Map / Stack Map

GC safe point 的栈帧可能是：

```text
FP+0   return address
FP-4   local i:int
FP-8   local obj:pointer
FP-12  local tmp:int
FP-16  local arr:pointer
register r1: int
register r2: pointer
```

pointer map 写：

```text
stack pointer slots: FP-8, FP-16
register pointer slots: r2
```

GC 只把这些位置作为 roots。

如果漏掉 `obj`：

```text
活对象可能被回收。
```

如果把 `i` 当 pointer：

```text
可能把垃圾错误保留下来，甚至访问非法地址。
```

## Type Descriptor

GC 扫描 heap object 时，也要知道对象内部哪些字段是指针。

例子：

```text
record { int x; pointer next; string name }
```

descriptor bitmap：

```text
0 1 1
```

表示：

- 第 1 个字段不是指针。
- 第 2、3 个字段是指针，要继续追踪。

对象布局、OO class descriptor 和 GC descriptor 经常有关联：descriptor 不只服务动态派发，也可能服务 GC 扫描。

## 指针信息如何贯穿编译阶段

PPT 给出的主线：

| 编译阶段 | 对 GC 支持 |
|---|---|
| Type checking | 知道每个表达式/变量类型，是 pointer 信息来源 |
| IR / Temp generation | 给每个 temp 标记 POINTER 或 NON_POINTER |
| Stack spilling | temp spill 到栈时，frame slot 也要标记是否 pointer |
| Register allocation | temp 分配到物理寄存器后，pointer tag 要转移到对应 register |
| Code emission | 在每个 safe point 发出 `{return address -> pointer registers/slots}` 的 pointer map |

这说明：

```text
pointer-ness 是一个从前端流到后端的属性。
```

如果 register allocation 之后丢失了“哪个物理寄存器里是指针”，GC 就无法精确扫描 roots。

## Safe Point

safe point 是程序可以安全暂停并运行 GC 的点。

常见 safe points：

- allocation slow path。
- function call。
- loop back edge。
- runtime check。

为什么不为每条指令都生成 pointer map？

```text
live sets 每条指令都可能不同，map 太大。
```

所以实际系统只在有限 safe points 记录精确信息。

## Derived Pointer

derived pointer 指向对象内部，例如：

```text
base = arr
p = &arr[i]
```

`p` 不是对象开头地址，而是对象内部地址。

如果 copying GC 移动 `arr`：

```text
只知道 p 不够，还要知道 base arr，才能重新计算 p。
```

考试只需知道：derived pointer 是 moving GC 和 precise GC 的难点之一。

## 弱化内容速记

### Generational GC

经验假设：

```text
Most objects die young.
```

做法：

- 堆分年轻代、老年代。
- 频繁回收年轻代。
- 较少回收老年代。

问题：

```text
老对象可能指向新对象。
```

需要：

- remembered set。
- write barrier。

### Incremental GC

目标：

```text
把一次长 GC 拆成很多小步，降低暂停时间。
```

常用 tricolor abstraction：

| 颜色 | 含义 |
|---|---|
| white | 尚未发现，最后仍 white 则可回收 |
| gray | 已发现，但子指针还没扫描完 |
| black | 已发现，且子指针已扫描 |

难点：程序和 GC 交错运行时，对象图会变化，需要 barrier 维护不变量。

## 判断题和选择题高频点

| 说法 | 判断 | 解释 |
|---|---|---|
| GC 回收所有以后不会再用的对象 | 不精确 | 实际按 reachability 保守近似 |
| 从 roots 不可达的对象可以回收 | 对 | tracing GC 的基本判断 |
| mark-and-sweep 能回收循环垃圾 | 对 | 基于 reachability |
| mark-and-sweep 不移动对象 | 对 | 因而可能有碎片 |
| reference counting 能回收所有 unreachable objects | 错 | 循环垃圾计数非零 |
| reference counting 每次指针赋值可能有额外开销 | 对 | 旧目标减、新目标加 |
| copying collection 能消除 external fragmentation | 对 | 活对象复制到连续 to-space |
| copying collection saves memory | 错 | 通常需要 semi-space 额外空间 |
| copying collection 可 fast allocation | 对 | bump pointer |
| copying GC 后不需要更新 roots | 错 | roots 和 fields 都要更新到新地址 |
| 精确 GC 需要 compiler pointer map | 对 | runtime 需要知道 roots |

## 常见误区

- “不可达”不等于理论上的“以后绝不会用”，但不可达对象可安全回收。
- Reference counting 中计数非零不代表从 roots 可达。
- Copying GC 必须更新所有 roots 和 pointer fields，否则指向旧 from-space。
- Copying GC 的优点是 compact 和 fast allocation，不是省内存。
- GC 不是只扫描堆，还必须从 roots 开始。
- 编译器如果不给 pointer map，精确 GC 很难安全工作。
- type descriptor 描述 heap object 内部字段，pointer map 描述 stack/register roots，不要混淆。
- generational/incremental 在本课只需速记作用和关键词。
